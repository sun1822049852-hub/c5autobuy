# AGENTS.md

本文件定义本仓库协作偏好与沟通约束；不覆盖系统安全规则与更高优先级指令。

## Agent 模型限定（特殊硬约束，必选）

- 默认仅使用 `gpt-5.2`、`gpt-5.3-codex`、`gpt-5.4`（含同代 codex 变体）。
- 禁止使用任何 `gpt-5.1*` 模型（例如 `gpt-5.1-codex-max`、`gpt-5.1-codex-mini`），除非用户在当前会话明确授权。
- 在可满足任务质量前提下，优先选择同代中更轻量模型处理机械性子任务，以控制用量。
- 上述模型限定属于特殊硬约束，优先级高于默认模型选择偏好与一般调度习惯。

## 流程模块

### 持续开启（强制）

- `会话日志 + 记忆文件`：
  - 会话日志文件固定为 `docs/agent/session-log.md`，按时间顺序追加，记录“做了什么 / 做到哪里 / 下一步”。
  - 记忆文件固定为 `docs/agent/memory.md`，仅沉淀跨会话稳定约束、长期决策和明确禁忌。
  - 防遗漏检查：
    - 只要发生仓库改动，必须追加或更新当次会话日志条目。
    - 若形成新的稳定约束，必须同步提炼到记忆文件。
    - 交付前必须确认两份文件与本次改动一致。

### 可选模块（按项目择优启用）

- `README 同步维护`：发生仓库改动后同步核对并维护 `README.md`；若无需改动，在交付说明中明确“README 已核对，本次无需改动”。
- `压缩前会话纪要`：准备会话压缩/收口前先写“已完成、当前位置、未完成、下一步”，并优先落到 `README.md` 协作记录区或等效记录文档。

## 打包纪律

- 非经用户在当前会话主动明确指定，不得为了常规验证、顺手确认或默认收口而重新生成安装包（例如执行 `npm --prefix app_desktop_web run build:win`）。
- 若只需验证发行资源、体积或 `win-unpacked` 结构，优先使用 `pack:win`、现有产物验尸或更小粒度检查，不默认升级到 installer 重建。

## 控制台现网同步纪律

- `program_admin_console/tools/connectProgramAdminConsole.{ps1,cmd}` 命中的始终是远端运行中的 `c5-program-admin`，不是本地工作树页面。
- 任何会影响 `program_admin_console` 远端控制台实际行为的改动，若目标是改变当前现网，就必须在同一任务内完成：
  - 远端源码同步；
  - 远端容器重建/替换；
  - 最小 smoke 验证。
- 最小 smoke 至少包含：
  - `http://127.0.0.1:18787/api/health`
  - `http://127.0.0.1:18787/api/admin/session`
  - `/admin` 对应静态资源是否已切到本地新版本（至少核对 `app.js` / `styles.css`，必要时比对哈希）
- 若本轮只做本地实验、尚未同步远端，必须明确标注“本地已改，远端未同步”，不得把本地行为写成“当前现网口径”。
- 远端热修不得只留在服务器上；凡已影响现网的控制台修复，必须回写仓库与文档，避免再次出现“本地和 connect 入口长期漂移”的情况。

## 变更前的“确认计划式”沟通（适用于会改变 UI / 交互 / 架构的修改）

在开始实现前，默认先通过聊天把理解收敛到可执行的计划，至少包含：

- 复述对需求与目标的理解（以产品行为与用户路径为主）。
- 列出“将要修改的点”（UI/交互/信息结构/模块职责/影响范围）。
- 明确“不会修改的点”（刻意不碰的页面、能力边界、兼容性范围）。
- 至少确认 1 个关键分歧或不确定点（例如：期望的交互、权限/校验、数据来源、回退策略）后再动工，避免跑偏。

## 沟通表达优先级

- 假设用户不需要理解代码实现细节，且通常看不懂文件路径/源码片段；用户更懂产品结构与架构取舍。
- 与用户沟通时，优先使用：交互行为、信息结构、模块职责、影响范围、边界条件与权衡。
- 不以文件清单、函数名、代码术语作为主要沟通载体；必要时仅作为补充。
- 解释链路、性能、调度或并发时，默认按“这一层负责什么、为什么存在、会不会增加等待、能否被上/下游吸收”来表述，不按函数调用顺序报菜名。
- 若用户明确表示看不懂代码，则后续回答中默认避免连续抛出函数名、线程名、类名；必须引用时，先说人话作用，再补代码标识。
- 若已有足够证据把问题直接归因到单一现象（例如看错窗口、未切换页面/筛选、旧窗口保留旧状态），默认先直接指出问题本身，再补证据；不要在证据已充分时先铺多种可能性或先给修复方案。

## 需要用户确认时的确认对象

- 优先让用户确认“产品行为与界面结果”（用户看得到/点得到的效果、校验与提示、数据展示与更新时机）。
- 除非用户主动要求，否则不要把“代码实现手段/实现路径”作为主要确认对象。

## 重复性环境坑处理规则

- 遇到 backend pytest 在导入阶段因可选依赖缺失而失败时，不得优先重复做单次命令临时注入或同类试错。
- 当前项目中，`xsign` 缺失属于已确认的重复性环境坑；若出现 `ModuleNotFoundError: xsign` 或等价导入错误，必须先检查：
  - `tests/backend/conftest.py`
  - 项目中的统一测试脚本（如果已提供）
  - 现有测试文件是否已有最小 stub / fallback 模式
- 在完成上述检查前，不得继续堆叠新的会话级临时注入命令。
- 同一类环境/依赖/测试启动错误若出现两次以上，必须沉淀为仓库规则、测试入口兜底或统一脚本，而不是继续依赖单次对话中的临时修补。

## 主链路性能红线

- `查询 -> 命中 -> 购买` 属于本项目核心主链路。
- 任何可能拖慢该主链路时延、吞吐、命中后下单速度或稳定性的修改，实施前必须先上报并获得确认，不得直接改动。
- 上报时至少说明：
  - 影响环节（查询、命中分发、下单、支付、状态回写）。
  - 额外开销或潜在性能损失来源。
  - 是否存在不影响主链路的替代方案。
  - 回退策略与验证方式。

## 关键行为保护（强制）

- 下列行为默认按关键行为处理，不得在缺少显式说明与验证证据时静默改写：
  - 登录成功链路。
  - 白名单 / open-api 打开与复用链路。
  - `查询 -> 命中 -> 购买` 主链路。
  - 关键页面可用性。
  - 既有数据获取链路。
  - 既有数据保存、回写与持久化链路。
- 命中关键行为时，实施前至少要先明确四件事：
  - 这次准备改什么。
  - 这次刻意不改什么。
  - 用什么验证证明行为没坏。
  - 若结果不对，如何回退。
- 只要改动会影响以下任一项，默认视为命中“数据获取和保存”关键行为：
  - 数据来源、筛选条件、字段映射、聚合口径。
  - 写库/写文件/写缓存/写会话的时机、目标位置、覆盖顺序。
  - 浏览器 session/profile/cookie 的复用、克隆、持久化与回填逻辑。
- 对关键行为，若当前仓库没有自动化测试，不得凭感觉静默改写；至少补最小回归测试，或明确写出人工验证路径与验证缺口。
- 不得把辅助动作反向绑死主成功链路。
  - 例如 profile 持久化、cookie 续期、缓存刷新、诊断写入失败，不能直接打断“登录成功”“白名单打开成功”“购买成功”这类主结果，除非产品明确要求如此。

## 新旧版本替换纪律（强制）

- 除非魔尊在当前会话明确要求兼容、灰度、迁移、回滚或保留调试隔离口径，否则实现新版时默认必须删除旧版路径，不得保留静默 fallback。
- 这里的“旧版路径”包括但不限于：
  - 旧 UI / 旧交互分支
  - 旧接口调用链
  - 旧配置开关默认分支
  - 旧启动/路由分发逻辑
  - 仅靠条件判断残留、可能被 stale 状态重新命中的兼容代码
- 若因兼容、迁移、灰度、debug 等原因必须暂时保留旧版，实施前至少要同时明确：
  - 为什么保留
  - 哪些入口仍会命中
  - 计划何时删除
  - 用什么验证证明不会意外回退到旧行为
- 不允许以“先留着以防万一”为理由长期并存双轨实现；若确需双轨并存，必须把它视为显式例外，而不是默认工程习惯。

## 桌面启动链路保护（强制）

- `main_ui_node_desktop.js`、`main_ui_node_desktop_local_debug.js` 与 embedded Python backend 的启动/接管链路，默认按关键可用性行为处理；未给出验证证据前，不得静默改写 ready 判定、deferred startup 装配顺序或 local debug 启动口径。
- Electron 侧把 backend 视为 `ready` 的前提，必须是 `/health` 明确返回 `ready=true`；单纯 HTTP 200 只表示进程活着，不代表 `/app/bootstrap` 等业务接口已可接管。
- deferred startup 所需 middleware 必须在 `create_app()` 阶段预先注册；不得在 lifespan 或 `_deferred_init()` 内再调用 `add_middleware()`，避免再次触发 `Cannot add middleware after an application has started`。
- 若改动触及桌面启动链、backend readiness gate、`/health` 语义、或 local debug 入口，收口前至少要补/跑对应 focused regression，默认包括：
  - `app_desktop_web/tests/electron/python_backend.test.js`
  - `tests/backend/test_backend_health.py`
  - `app_desktop_web/tests/renderer/app_remote_bootstrap.test.jsx`

## 前端预览默认口径

- 只要当前任务的目标是“看页面 / 改 UI / 调布局 / 按页面结果改前端”，且页面可通过 `localhost`、独立 dev server 或 file-backed preview 打开，默认优先使用 Codex app 的 in-app browser 预览页面，再推进评论、截图、验收与改动；不得先入为主默认拉起 Electron 壳或程序自带前端。
- 若当前环境是 CLI、或桌面 app 当下没有 in-app browser / Annotating 能力，也应先走 `web-only dev server + 页面 URL/截图` 这条轻量链路；在未证明必须依赖桌面壳之前，不得直接升级为启动完整程序壳。
- 只有命中以下任一条件时，才默认切回程序自带前端、Electron 主入口或 embedded 桌面链路：
  - 任务直接涉及窗口、托盘、菜单、原生 API、桌面壳交互。
  - 任务直接涉及 embedded backend 接管、program access、桌面 bootstrap、packaged release 或桌面启动链。
  - 任务必须复用真实登录态、cookie、profile、浏览器扩展或系统浏览器环境。
  - 问题已确认只在程序壳内复现，浏览器预览口径无法覆盖。
- 若用户只说“打开前端”而未额外限定，默认先理解为“打开可供 Codex 预览与修改的浏览器页面”，优先提供可被内置浏览器使用的页面口径。
- 交付时必须明确写清本轮验证口径：若只在 in-app browser / web 预览里验过，就标注“浏览器预览已验证，程序壳未验证”；若只验了程序壳，也要显式写明“程序壳已验证，浏览器预览未复验”，避免把两个口径混写。
