# CLAUDE.md

本文件定义本仓库在 Claude Code / Kiro 下的协作偏好与约束。内容与 `AGENTS.md`（Codex 侧）对齐，确保跨工具一致性。

## 用户偏好（硬约束）

- **用户不操作服务器**：所有服务端验证（启动服务、重启、curl 测试、端口检查等）必须由 agent 在本地完成，不可要求用户手动操作。变更后必须自行启动服务并验证效果。
- 用户不需要理解代码实现细节，更懂产品结构与架构取舍。沟通时优先使用交互行为、信息结构、模块职责、影响范围、边界条件与权衡，不以文件清单、函数名、代码术语作为主要沟通载体。
- 需要用户确认时，优先让用户确认"产品行为与界面结果"，不要把代码实现手段作为主要确认对象。

## 变更前确认（适用于 UI / 交互 / 架构修改）

在开始实现前，默认先收敛到可执行的计划：

- 复述对需求与目标的理解（以产品行为与用户路径为主）。
- 列出"将要修改的点"与"不会修改的点"。
- 至少确认 1 个关键分歧或不确定点后再动工。

## 打包纪律

- 非经用户在当前会话主动明确指定，不得为了常规验证而重新生成安装包（如 `npm --prefix app_desktop_web run build:win`）。
- 验证优先使用 `pack:win`、现有产物检查或更小粒度手段。

## 主链路性能红线

- `查询 -> 命中 -> 购买` 属于核心主链路。
- 任何可能拖慢该主链路时延、吞吐、命中后下单速度或稳定性的修改，实施前必须先上报并获得确认。
- 上报时至少说明：影响环节、额外开销来源、替代方案、回退策略与验证方式。

## 关键行为保护（强制）

下列行为不得在缺少显式说明与验证证据时静默改写：

- 登录成功链路
- 白名单 / open-api 打开与复用链路
- `查询 -> 命中 -> 购买` 主链路
- 关键页面可用性
- 既有数据获取链路
- 既有数据保存、回写与持久化链路

命中关键行为时，实施前至少明确：改什么、不改什么、怎么验证、怎么回退。

只要改动影响数据来源/筛选条件/字段映射/聚合口径、写库写文件写缓存写会话的时机与目标、浏览器 session/profile/cookie 的复用与持久化逻辑，默认视为命中关键行为。

不得把辅助动作反向绑死主成功链路。

## 重复性环境坑

- `xsign` 缺失属于已确认的重复性环境坑；遇到时先检查 `tests/backend/conftest.py`、统一测试脚本、现有 stub/fallback，不得堆叠临时注入。
- 同类环境错误出现两次以上，必须沉淀为仓库规则或自动兜底。

## 已确认的产品约束

- 配置管理页：`market_price` 不可手动修改，值来自自动查询。
- 程序品牌统一为 `C5 交易助手`。
- 程序会员控制台管理员登录失败文案固定为"用户名或者密码错误"。
- 管理员密码以 scrypt 哈希保存，遇登录失败不要尝试查看原密码，优先确认用户名或执行密码重置。

## 项目结构速查

- `app_desktop_web/` — Electron 桌面端
- `app_backend/` — Python 后端
- `program_admin_console/` — 管理控制台（Node.js http + SQLite + vanilla JS）
  - 后端：`src/server.js`、`src/controlPlaneStore.js`
  - 前端：`ui/index.html`、`ui/app.js`、`ui/styles.css`
  - 测试：`npm --prefix program_admin_console test`
- `AGENTS.md` — Codex 侧协作配置（与本文件对齐）
