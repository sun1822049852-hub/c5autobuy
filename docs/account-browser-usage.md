# 账号浏览器使用说明

## 1. 这是什么

本系统为每个账号维护一套独立的浏览器 Profile。

可以把它理解为：

- 一个账号 = 一套单独的浏览器登录环境
- 登录成功后，登录状态会保存在该账号自己的浏览器环境里
- 后续再次打开相关页面时，会复用这套环境，而不是重新新建一个空白浏览器

这套浏览器主要用于：

- 登录 C5 账号
- 复用登录状态打开 API 绑定页
- 获取并同步 API Key、白名单 IP、当前公网 IP 等信息

它不是“普通用户手动双击打开的本机浏览器”，而是“系统管理的账号业务浏览器环境”。

---

## 2. 使用原则

请先记住这 4 条：

1. 每个账号都有自己独立的浏览器环境，账号之间不会共用登录状态。
2. 第一次必须先登录，登录完成后系统才会保存该账号的浏览器 Profile。
3. 后续点击“添加白名单”时，系统会复用该账号之前登录成功的浏览器 Profile。
4. 如果登录态失效，或者提示缺少可复用登录会话，就需要重新登录一次。

---

## 3. 浏览器环境说明

这套浏览器环境由 3 部分组成：

- 受管浏览器 Runtime
- 账号独立 Profile
- 启动时生成的临时会话副本

可以这样理解：

- `受管浏览器 Runtime`：程序使用固定的浏览器运行环境，避免依赖用户机器上的默认浏览器状态。
- `账号独立 Profile`：每个账号都有自己独立的登录环境、Cookie、站点状态。
- `临时会话副本`：每次拉起浏览器时，系统会从该账号的 Profile 复制出一份临时会话来运行，关闭后再回写，避免不同账号互相污染。

所以它的本质不是“直接操作系统默认浏览器”，而是“按账号管理的一套业务浏览器环境”。

---

## 4. 关于 Selenium / 受控特征

这套链路本质上仍然属于受控浏览器链路。

原因是系统需要在登录或打开绑定页后继续完成这些动作：

- 识别当前账号是否已经登录
- 读取 Cookie
- 同步 API Key
- 同步白名单 IP
- 同步当前公网 IP

为了完成这些能力，当前实现复用了原登录流程的技术路线，因此会带有自动化控制特征。

需要明确的是：

- 使用 Selenium / remote debugging / 受控启动方式，在一些网站上可能会被识别为开发者模式或自动化环境。
- 这套浏览器不应理解为“反检测浏览器”或“纯手工自然浏览器”。
- 当前项目之所以继续采用这套方式，是因为原登录链和原信息获取链已经验证可用，业务依赖也建立在这条链上。

因此，对外说明时应把它定义为：

`一个可复用账号登录态、可自动同步关键信息的系统管理浏览器环境`

而不是：

`完全等同于人工自然使用的普通浏览器`

---

## 5. 标准使用流程

### 第一步：新增账号

先在账号中心新增一个账号，并按需要填写：

- 浏览器代理
- API 代理
- API Key（如需要手动维护）

如果浏览器走代理，那么后续浏览器打开页面时也会沿用该浏览器代理。

---

### 第二步：点击“登录”

点击账号的“登录”后，系统会拉起该账号对应的浏览器环境。

此时请在打开的浏览器中完成正常登录操作。

登录成功后，系统会自动保存：

- 该账号的登录 Cookie
- 该账号的浏览器 Profile
- 该账号关联的 API 信息

同时系统会尝试同步：

- API Key
- 白名单 IP
- 当前公网 IP

---

### 第三步：登录完成后关闭浏览器

当账号信息已经成功显示后，可以关闭这次登录时拉起的浏览器窗口。

关闭后不会丢失该账号的浏览器会话，因为登录状态已经保存在该账号自己的 Profile 中。

---

### 第四步：点击“API IP 设置”里的“添加白名单”

需要修改白名单时：

1. 打开账号的 `API IP 设置`
2. 点击 `添加白名单`

系统会做的事情是：

- 复用这个账号之前保存的浏览器 Profile
- 按该账号当前的浏览器代理方式启动浏览器
- 直接打开 C5 的 API 绑定页

这一步不是重新登录，也不是新建空白浏览器，而是继续使用该账号原来的登录环境。

---

### 第五步：在打开的页面中手动处理白名单

浏览器打开绑定页后，由使用者自己在页面中完成：

- 添加白名单 IP
- 修改白名单 IP
- 确认 API 绑定页内容

操作完成后，关闭该浏览器窗口即可。

系统后台会继续基于该账号当前浏览器会话同步：

- API Key
- 白名单 IP
- 当前公网 IP

---

## 6. 适合怎样理解这套浏览器

最简单的理解方式是：

它不是“每次临时拉起一个新浏览器”，而是“给每个账号配一套专属浏览器用户配置”。

所以它更接近于：

- Chrome / Edge 的多用户配置
- 同一台机器上不同浏览器用户

区别只是这些用户配置由系统自动按账号维护。

---

## 7. 什么情况下要重新登录

遇到以下情况时，需要重新登录：

- 点击“添加白名单”时提示缺少可复用登录会话
- 浏览器打开后发现 C5 已退出登录
- API Key、白名单、账号身份长时间无法同步
- C5 网站主动让当前登录态失效

重新登录后，系统会刷新这个账号自己的浏览器 Profile。

---

## 8. 使用时的注意事项

### 不要做的事

- 不要手动删除程序目录下的浏览器 Profile 文件
- 不要把一个账号的 Profile 拿去给另一个账号混用
- 不要认为关闭窗口就等于清空登录态
- 不要把这套浏览器当成“绝对不会被识别的纯净浏览器”
- 不要把它用于与本项目无关、且强依赖反自动化规避的场景

### 建议做的事

- 一个账号固定使用一个备注名，便于识别
- 修改代理后，重新检查 IP 与白名单是否匹配
- 白名单变更后，观察账号中心里的 API 信息是否已同步
- 需要对外说明时，明确告知这是“系统管理的账号浏览器环境”，不是普通手工浏览器

## 9. 常见问题

### 1）为什么登录一次后，后面还能继续打开绑定页？

因为登录成功后，系统已经把该账号的浏览器 Profile 保存下来了，后续会直接复用。

### 2）为什么不同账号不会串号？

因为每个账号使用的是独立的浏览器 Profile，登录状态彼此隔离。

### 3）为什么有时必须重新登录？

因为网站侧登录态不是永久有效的。即使本地 Profile 还在，服务端也可能让登录失效。

### 4）代理会不会影响这个浏览器？

会。浏览器启动时会按该账号当前配置的浏览器代理方式运行。

### 5）为什么不能把它理解成普通人工浏览器？

因为它仍然沿用了原登录流程和原信息同步流程，需要系统在浏览器会话上继续完成登录识别、信息抓取和状态同步，所以它带有受控浏览器特征。

### 6）既然有受控特征，为什么还继续用？

因为当前项目的原登录链和原信息获取链已经验证可用，系统现有的 Cookie、API Key、白名单、IP 同步能力都依赖这条链。当前方案不是为了伪装成普通浏览器，而是为了稳定复用账号会话并持续同步业务信息。

---

## 10. 一句话版本

这套浏览器的使用方式可以概括为：

先登录一次，保存该账号自己的浏览器 Profile；以后打开白名单页面时，直接复用这个账号原来的业务浏览器环境，不需要重新登录，只在登录态失效时重新登录。

## 11. 给开发者 / AI 的复现口径

如果后续需要让 AI 或其他开发者复现当前项目的浏览器行为，请直接按下面这套口径说明，不要换成另一套“看起来更像普通 Selenium”的写法。

### 核心原则

当前项目的主链路不是：

- `webdriver.Edge(...)`
- `driver.get(...)`
- 由 WebDriver 直接驱动页面完成登录与后续 API 页访问

当前项目实际采用的是：

1. 为账号准备独立的浏览器 Profile
2. 每次操作时先复制出一份临时 session
3. 用 `subprocess.Popen(...)` 直接启动真实 `msedge.exe`
4. 启动参数中带上：
   - `--remote-debugging-port`
   - `--user-data-dir`
   - `--profile-directory=Default`
5. 浏览器启动后，不走“WebDriver 驱页主链”，而是通过 CDP 附着读取会话、Cookie、页面状态和接口数据
6. 登录成功后把 session 回写到账号 Profile，后续继续复用

也就是说，这个项目的真实浏览器使用方式应理解为：

`账号级 Profile 复用 + 真 Edge 进程启动 + CDP 附着读取`

而不是：

`Selenium 直接创建一个受控 driver 再拿它去跑完整页面流程`

### 开发时必须保持的行为

如果 AI 要复现当前逻辑，必须遵守下面这些约束：

- 登录时：复制账号 Profile，启动真实 Edge，等待用户手工完成登录，再通过 CDP 读取登录结果
- 打开“添加白名单”时：复用该账号已保存的 Profile，再启动真实 Edge 直接打开 `https://www.c5game.com/user/user/open-api`
- 如果浏览器代理存在：启动浏览器时沿用该代理
- 如果浏览器代理是 `direct` 或为空：按直连方式启动
- 读取账号信息时：沿用现有 CDP / 页面状态读取逻辑，不要另起一套全新的 WebDriver 页面点击脚本
- 如果 AI 提议“直接新建一个 webdriver 然后 get 页面处理”，应视为偏离当前项目实现

### 当前项目里的真实落点

当前主链关键文件如下：

- `app_backend/main.py`
  - 主入口把登录链路绑定为 `ManagedEdgeCdpLoginRunner(...).run`
  - 主入口把“添加白名单”页面打开逻辑绑定为 `OpenApiBindingPageLauncher(...)`
- `app_backend/infrastructure/browser_runtime/login_adapter.py`
  - `ManagedEdgeCdpLoginRunner` 负责复制 session、启动真实 Edge、等待登录完成
- `app_backend/infrastructure/browser_runtime/open_api_binding_page_launcher.py`
  - `OpenApiBindingPageLauncher` 负责复用账号 Profile 打开 API 绑定页
- `app_backend/infrastructure/browser_runtime/cdp_session_reader.py`
  - 负责从已附着的浏览器会话里读取 Cookie、账号信息、API 页面状态

### 为什么现在要按 browser_runtime 理解项目

当前活跃目录已经收口到 `app_backend/infrastructure/browser_runtime/`，这里表达的是：

1. 管理真实 Edge 进程、账号 Profile、临时 session 与 CDP 附着
2. 读取真实浏览器中的 Cookie、页面状态与 API 白名单信息
3. 复用登录后的账号环境，而不是重新造一条自动化驱页链

当前生产主链里，真正起作用的是：

- 真实 Edge 进程启动
- 账号 Profile 复用
- CDP 附着读取

不是：

- 新建一个干净的 `webdriver.Edge()` 后靠自动化脚本把整条业务链跑完

### 可以直接发给 AI 的复现说明

下面这段话可以直接发给 AI：

> 请按当前项目的真实浏览器链路复现，不要改成经典 Selenium WebDriver 驱页方案。现有实现是：账号独立 Profile 持久化，操作时先克隆临时 session，用 `subprocess.Popen(...)` 启真实 `msedge.exe`，参数包含 `--remote-debugging-port`、`--user-data-dir`、`--profile-directory=Default`，然后通过 CDP 附着读取 Cookie、登录态、API 页面状态。登录完成后回写 session；打开白名单页时复用该账号 Profile 直接打开 `https://www.c5game.com/user/user/open-api`。如果浏览器代理存在则沿用浏览器代理，如果是直连则按直连启动。不要另写一套 `webdriver.Edge().get(...)` 的页面自动化流程。
