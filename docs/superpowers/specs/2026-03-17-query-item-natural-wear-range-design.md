# 查询商品天然磨损范围字段设计

## 1. 目标

按旧版 `autobuy.py` 的真实语义，把“商品天然完整磨损范围”和“用户自己设置的查询阈值”分开保存，并在新架构里补齐这条链。

本次设计目标：

- 保留现有 `max_wear` 语义不变，继续表示用户接受的最大磨损阈值
- 新增独立字段保存商品天然最大磨损
- 新增商品时写入天然最大磨损
- 启动前详情刷新时更新天然最大磨损
- 前端详情展示和准备对话框显示完整磨损范围

本次不改查询接口参数，不改三种查询模式调度逻辑，不改购买逻辑。

## 2. 源代码逻辑锁定

旧版源码里有两套不同数据：

1. 数据库 `items` 表保存商品天然详情：
   - `minwear`
   - `maxwear`
   - `minPrice`
2. 配置里的 `ProductItem` 保存用户扫货参数：
   - `minwear`
   - `max_wear`
   - `max_price`
3. 新增商品时：
   - 自动把 `db_minwear` 作为当前配置 `minwear`
   - 只让用户输入“最高接受磨损”
   - 当前配置的 `max_wear` 是用户阈值，不是商品天然 `maxwear`
4. 查询执行器发请求时使用的是当前配置的 `minwear / max_wear`
5. 商品详情刷新更新的是商品天然范围，不覆盖用户阈值

所以新架构如果只有一个 `max_wear`，语义就不完整。

## 3. 当前缺口

当前新架构的 `QueryItem` 只有：

- `min_wear`
- `max_wear`

其中：

- `min_wear` 当前承担“天然最小磨损 / 自动采用的最佳最小磨损”
- `max_wear` 当前承担“用户查询阈值”

但缺少旧版数据库里的天然最大磨损字段，因此：

- 无法展示完整磨损范围
- 刷新时无法同步天然最大磨损
- 准备对话框无法复刻旧版“完整磨损范围”信息

## 4. 方案比较

### 方案 A：新增 `detail_max_wear` 字段，保留 `max_wear`

优点：

- 与旧版语义一一对应
- 查询执行链不需要改语义
- 刷新链和展示链都能补完整

缺点：

- 需要补数据库、仓储、接口、前端展示和测试

### 方案 B：把现有 `max_wear` 改成天然最大磨损，再另外找地方存用户阈值

优点：

- 表面上也能分离两种语义

缺点：

- 会破坏当前查询执行器已经接入的 `max_wear` 语义
- 影响面更大，回归风险更高

### 方案 C：不扩字段，只在前端临时拼显示

优点：

- 后端改动少

缺点：

- 数据模型继续不完整
- 刷新和新增都没有稳定存储
- 以后再接更多界面还得重复补洞

## 5. 结论

采用方案 A：

- 新增 `detail_max_wear`
- 保留 `max_wear` 继续表示用户阈值

这是最贴近旧源码、风险最小、边界最清楚的方式。

## 6. 新设计

### 6.1 数据模型

在 `QueryItem`、数据库表、API schema 中新增：

- `detail_max_wear: float | None`

字段语义：

- `min_wear`: 当前配置使用的最小磨损，按旧逻辑等于商品天然最小磨损
- `detail_max_wear`: 商品天然完整磨损范围的最大值
- `max_wear`: 用户接受的最大磨损阈值

### 6.2 新增商品流程

新增商品时通过 `ProductDetailCollector` 获取：

- `min_wear`
- `max_wear`（collector 返回的天然最大磨损）
- `last_market_price`

落库时映射为：

- `min_wear = detail.min_wear`
- `detail_max_wear = detail.max_wear`
- `max_wear = 用户输入阈值`

这样保持旧版“自动最小磨损 + 用户输入最大阈值”的逻辑。

### 6.3 启动前刷新流程

刷新服务继续只更新商品详情字段，不覆盖用户阈值。

刷新成功后更新：

- `item_name`
- `market_hash_name`
- `min_wear`
- `detail_max_wear`
- `last_market_price`
- `last_detail_sync_at`

明确不更新：

- `max_wear`
- `max_price`

### 6.4 API 返回

`QueryItemResponse`、`/query-runtime/prepare` 返回项补充：

- `detail_max_wear`

这样前端详情页和准备对话框都能直接显示完整范围。

### 6.5 前端展示

详情面板：

- 原“磨损上限”列改为更明确的展示
- 同时显示：
  - 完整磨损范围：`min_wear ~ detail_max_wear`
  - 用户阈值：`max_wear`

准备对话框：

- 增加完整磨损范围列
- 保留最低价、状态、说明

避免把“天然范围”和“用户阈值”混成一个字段。

## 7. 错误处理

- 如果 collector 没返回天然最大磨损，则 `detail_max_wear` 允许为空
- 旧数据没有该字段时，前端显示为空，不阻塞查询
- 刷新失败时仍保留原值，不清空字段

## 8. 测试重点

必须锁定这些行为：

1. 新增商品时，`detail_max_wear` 保存 collector 返回的天然最大磨损
2. 用户阈值 `max_wear` 仍按原逻辑保存
3. 刷新商品详情时更新 `detail_max_wear`
4. 刷新不覆盖用户阈值 `max_wear`
5. 配置详情接口会返回 `detail_max_wear`
6. 前端详情和准备对话框显示完整磨损范围

## 9. 非目标

本次不做：

- 重新设计查询参数结构
- 修改三种查询执行器的请求参数
- 改造商品最小磨损语义
- 做数据库迁移工具；当前仍沿用测试和本地新库自动建表方式

## 10. 结论

这次改动的核心就是把旧版一直存在、但当前新架构遗漏的语义补回来：

- `detail_max_wear` 是商品天然范围
- `max_wear` 是用户阈值

这样后续把 `autobuy.py` 彻底删掉时，查询配置的数据语义才算真正对齐。
