# Login Runtime Replacement Design

## 1. 目标

在不改变现有登录交互、任务状态流、冲突处理和账号持久化对外行为的前提下，把登录底层执行链路从 `autobuy.py` 的 legacy `SeleniumLoginManager` 中剥离出来，接入新架构自己的 Selenium 登录运行器。

本次只替换：

- 浏览器启动与代理配置
- 登录页监控脚本注入
- 扫码成功判定
- 用户信息提取
- cookie 提取与 `NC5_deviceId` 更新

本次明确不替换：

- 登录任务入口
- 登录任务状态流
- 冲突处理逻辑
- 账号保存与覆盖/新增策略
- 前端登录对话框与交互

## 2. 约束

这次替换必须遵守以下硬约束：

- 不改前端登录体验
- 不改后端登录 API
- 不改 `LoginCapture`
- 不改登录任务状态名称和顺序
- 不改冲突处理 payload 结构
- 不改“用户扫码、浏览器自行抓取信息、用户手动关闭浏览器后任务完成”的现有流程

也就是说：

- 对外合同不变
- 只替换登录执行内核

## 3. 当前问题

当前登录链路虽然已经有新的任务入口、任务管理、冲突处理和账号持久化，但在设计时真正执行登录时仍然依赖：

- `SeleniumLoginAdapter`
- `autobuy.py` 中的 `SeleniumLoginManager`

这带来几个问题：

1. 登录底层仍依赖超大 legacy 文件
2. 浏览器启动、监控脚本和 cookie/deviceId 处理逻辑无法在新架构内独立维护
3. 只要登录底层还未迁出，`autobuy.py` 就无法彻底删除

## 4. 方案比较

### 方案 A：一比一迁移登录执行内核

- 新增一个新的 Selenium 登录运行器
- adapter 对外接口保持不变
- 运行器内部尽量按旧逻辑直译迁移

优点：

- 风险最小
- 最符合“流程不变，只去 legacy 依赖”的目标
- 最容易做回归验证

缺点：

- 第一版内部结构不会特别漂亮

### 方案 B：迁移时同步拆成多个细模块

- 新增登录运行器
- 再拆浏览器启动、代理插件、监控脚本、数据提取等多个模块

优点：

- 长期结构更清晰

缺点：

- 当前收益不大
- 很容易把稳定流程拆坏

### 方案 C：重写成另一套登录方案

- 不再沿用当前监控脚本与 Selenium 流程

优点：

- 理论上可以更统一

缺点：

- 风险最大
- 明显偏离“保留原交互流程”的要求

## 5. 结论

采用方案 A。

原因：

- 风险最小
- 最符合“最终删除 `autobuy.py`”的目标
- 能在不改变用户体验的前提下，先把登录底层从 legacy 中拔出来

## 6. 新设计

### 6.1 登录任务边界

保持不变：

- `StartLoginTaskUseCase`
- `run_login_task(...)`
- `ResolveLoginConflictUseCase`

这些模块继续负责：

- 发起登录任务
- 转发任务状态
- 冲突处理
- 保存账号

### 6.2 新增 Selenium 登录运行器

新增一个新的 Selenium 登录运行器。

职责：

- 启动 Edge 浏览器
- 按账号代理配置设置代理
- 注入监控脚本
- 打开 C5 登录页
- 等待扫码成功
- 捕获用户信息和 cookie
- 等待用户手动关闭浏览器
- 返回登录结果

它不负责：

- 账号保存
- 冲突判定
- 任务持久化
- GUI 展示

### 6.3 `login_adapter`

保持对外合同不变：

- 继续暴露 `run_login(proxy_url=..., emit_state=...)`
- 继续返回 `LoginCapture`

变化仅在内部：

- 不再加载 `autobuy.py`
- 改为调用新的 Selenium 登录运行器

## 7. 数据流

### 7.1 调用路径

1. `StartLoginTaskUseCase` 创建登录任务
2. `run_login_task(...)` 调用 `login_adapter.run_login(...)`
3. 新 adapter 调用新的 Selenium 登录运行器
4. 运行器完成浏览器操作并返回登录结果
5. adapter 继续返回 `LoginCapture`
6. `run_login_task(...)` 继续按当前逻辑更新账号或进入冲突处理

### 7.2 任务状态流

必须保持不变：

- `starting_browser`
- `waiting_for_scan`
- `captured_login_info`
- `waiting_for_browser_close`
- `saving_account`

## 8. 必须保持的登录行为

### 8.1 登录交互

必须继续保持：

- Selenium 打开浏览器
- 用户手动扫码
- 浏览器自己抓取登录后的用户信息和 cookie
- 用户手动关闭浏览器
- 浏览器关闭后任务才完成

### 8.2 成功判定

必须继续保持：

- 以跳转到用户页作为主要成功判定
- 优先从监控脚本中捕获用户信息 API 响应
- 如果监控脚本没有数据，再走直接提取兜底

### 8.3 cookie 处理

必须继续保持：

- cookie 优先从捕获数据中获取
- 如果捕获数据没有 cookie，再从浏览器 cookies 拼接

### 8.4 `NC5_deviceId` 规则

这是本次迁移必须严格兼容的关键规则：

- 登录成功后，必须拿到 `NC5_deviceId`
- 如果拿不到 `NC5_deviceId`，直接判登录失败
- 拿到后，必须按旧规则重写一次 `NC5_deviceId`
- 新值规则：
  - `13位毫秒时间戳 + 5位随机数`
- 最终持久化的是“更新过 `NC5_deviceId` 的 cookie”

也就是说：

- 不是直接原样保存扫码得到的 cookie
- 而是先更新 `NC5_deviceId`，再保存

### 8.5 失败与取消

必须继续保持：

- 用户提前关闭浏览器且尚未完成登录，视为取消
- 登录超时，视为失败
- 无法提取用户信息，视为失败
- 无法拿到 cookie，视为失败
- 无法拿到 `NC5_deviceId`，视为失败

## 9. 测试策略

### 9.1 adapter 契约测试

至少覆盖：

1. adapter 继续返回 `LoginCapture`
2. 代理参数继续正确传入
3. 状态回调顺序不变
4. adapter 不再依赖 `autobuy.py`

### 9.2 登录任务流回归

至少覆盖：

1. 登录成功后仍能绑定购买能力并落库
2. 同账号登录仍更新当前账号
3. 不同账号登录仍进入 conflict
4. `create_new_account` 与 `replace_with_new_account` 行为不回归

### 9.3 运行器单测

至少覆盖：

1. 成功登录并拿到用户信息
2. 监控脚本无数据时走兜底提取
3. 用户提前关闭浏览器时返回取消
4. 登录超时返回失败
5. cookie 回退逻辑正确
6. `NC5_deviceId` 存在时会被重写
7. 缺失 `NC5_deviceId` 时直接失败
8. 代理参数正确传入浏览器配置

### 9.4 全量回归

至少覆盖：

1. 登录相关后端测试继续通过
2. 账号中心和前端登录任务相关测试继续通过
3. 全项目 pytest 继续通过

## 10. 非目标

本次不做：

- 改造登录流程为无浏览器模式
- 替换 Selenium 为 Playwright
- 优化前端登录 UI
- 调整冲突处理策略
- 重构账号保存模型

## 11. 结论

本次采用“登录任务流不动，只替换 Selenium 登录执行内核”的渐进方案：

- 风险最小
- 能保留当前稳定的扫码流程和冲突处理
- 能继续推进最终删除 `autobuy.py`
