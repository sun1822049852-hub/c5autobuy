# 查询工作台与商品分配器设计

日期：2026-03-19

## 1. 目标

为新的桌面化 Web UI 设计一套独立的“查询工作台”，用于替代当前旧查询页的只读详情模式，并补齐当前新后端尚未承载的商品级分配语义。

本次设计目标：

- 在新 UI 中支持多个查询配置的保存、切换与编辑
- 同一时刻只允许运行一个查询配置
- 在单个配置内管理多个商品的查询参数
- 支持商品级专属账号目标分配
- 支持共享池轮转分配
- 支持商品手动暂停
- 在运行时正确表达“专属 / 共享 / 无可用账号”等状态
- 保留并迁移旧 `autobuy.py` 的 ready 驱动查询分配思路

本次设计不是单纯的前端换皮，而是查询 UI 与查询运行时语义的联合设计。

## 2. 当前问题

当前系统存在以下缺口：

1. `app_desktop_web` 目前只有账号中心，查询系统仍未形成真实页面。
2. 旧 `PySide6` 查询页更接近“配置列表 + 只读详情”，不适合承载复杂商品编辑。
3. 当前后端只支持：
   - 配置基础信息
   - 商品级 `min_wear / max_wear / max_price`
   - 配置级 `mode_settings`
4. 当前后端不支持：
   - 商品级专属目标账号分配
   - 商品级手动暂停
   - 专属池 / 共享池的运行时分配状态
   - 账号失效后的自动降级与恢复
5. 旧 `autobuy.py` 里存在 ready 上报 + 商品指针轮转的查询分配器语义，但新后端目前只迁移了轻量商品调度器，还未补齐分配层语义。

## 3. 已确认的用户决策

以下内容已经在讨论中确认：

- 页面采用方案 A：
  - 左侧配置导航
  - 右侧当前配置工作台
  - 商品在工作台内行内展开编辑
- 多个配置可以保存，但同一时刻只能运行一个配置
- 右侧工作台右上角提供 `添加商品`
- 新增商品时：
  - 粘贴商品 URL
  - 自动拉取商品详情
  - 自动回填名称、默认磨损、价格参考
  - 三种模式目标分配数默认都为 `0`
- 商品行必须直接显示：
  - 商品名
  - 当前设置价格
  - 当前设置磨损
  - 三种模式目标分配数
  - 当前状态
- 商品状态按模式拆分展示，不做单一汇总态
- 保存时必须校验：
  - 每种模式下，商品专属目标分配总和不能超过当前可用账号数
- 三种模式的输入区需要动态提示“还可分配 X”
- 目标分配数表示该商品在该模式下的专属目标账号数
- `专属账号` ready 后只查询自己的绑定商品
- 未被专属占用的剩余账号进入共享池
- 共享池账号 ready 后，参考旧 `autobuy.py` 的商品指针轮转逻辑查询共享商品
- 某模式下目标分配数为 `0` 的商品，默认属于该模式的共享池候选商品
- 某商品只要当前还有至少 `1` 个专属账号，就不吃共享池
- 某商品专属账号实际数降到 `0` 后，进入共享池轮转
- 后续只要系统重新有足够账号满足它的目标分配数，就立刻退出共享池，回到专属态
- 这里不要求“原有专属账号恢复”，只看当前全局是否重新满足目标数
- 商品行状态字段只要求展示“目标分配数”和“当前状态”；当前状态可以包含实际分配数和其他状态
- 商品手动暂停仍然是需求，后端当前未支持
- 一个商品若在某模式下当前被多个账号查询，其最小可查询时间优化为：
  - `0.5 / 当前实际分配账号数量`

## 4. 方案比较

### 方案 A：左侧配置导航 + 右侧工作台 + 商品行内展开编辑

优点：

- 最贴合“多配置、单活动配置”的工作台语义
- 左侧负责切换配置，右侧负责深度编辑，职责清晰
- 商品总览和单商品细项可以同时兼顾
- 后续增加运行态状态条、启动停止区也更自然

缺点：

- 需要设计好右侧工作台密度，避免过于拥挤

### 方案 B：左侧配置手风琴展开

优点：

- 结构直观，点开即见内容

缺点：

- 配置数量一多会迅速变得拥挤
- 长列表与商品编辑区会互相争抢纵向空间

### 方案 C：左侧配置导航 + 中间商品表格 + 右侧商品抽屉

优点：

- 主表格最整洁
- 单商品编辑空间最大

缺点：

- 多商品连续编辑时来回切换成本较高
- 不如行内展开直观

### 结论

采用方案 A。

原因：

- 它最适合魔尊确认过的“多配置 + 单活动配置 + 商品行总览 + 行内展开编辑”工作方式。

## 5. 页面结构设计

### 5.1 页面总骨架

查询页分为左右两栏：

1. 左侧：配置导航区
2. 右侧：当前配置工作台

左侧负责选择配置，右侧负责编辑配置。

### 5.2 左侧配置导航区

结构：

- 标题：`查询配置`
- 按钮：`新建配置`
- 配置卡片列表

每个配置卡片显示：

- 配置名
- 商品数
- `未保存` 轻标签
- `运行中` 轻标签

点击配置卡片后：

- 不在左侧列表内部做手风琴展开
- 只切换右侧工作台内容

### 5.3 右侧当前配置工作台

结构：

1. 顶部配置摘要区
2. 中部商品列表区
3. 右侧新增商品面板入口
4. 底部固定操作条

#### 顶部配置摘要区

显示：

- 配置名称
- 配置描述
- 配置级状态

配置级状态包括：

- `未保存`
- `已保存`
- `运行中`
- `等待账号`
- `已停止`
- `校验失败`

#### 右上侧动作区

显示：

- `添加商品`

点击后：

- 在当前配置区域右侧展开新增商品面板
- 不打开独立弹窗
- 不离开当前工作台

#### 底部固定操作条

左侧显示保存反馈：

- `有未保存修改`
- `校验失败，无法保存`
- `已保存`

右侧显示：

- `保存`

后续如接入启动停止按钮，也应放在同一条底部操作条中。

## 6. 商品列表与编辑交互

### 6.1 商品主表格

商品表格默认每行只显示关键信息，避免页面一上来就被大量输入框淹没。

推荐列：

- 展开箭头
- 商品名
- 当前设置价格
- 当前设置磨损
- `new_api`
- `fast_api`
- `token`
- 当前状态

### 6.2 商品行字段语义

- `当前设置价格`：显示 `max_price`
- `当前设置磨损`：显示 `min_wear ~ max_wear`
- `new_api / fast_api / token`：显示该商品在对应模式下的专属目标账号数
- `当前状态`：按模式拆分展示运行时状态标签

### 6.3 商品状态展示

商品状态列不做单一汇总态，而是按模式拆成三枚紧凑标签。

示例：

- `new_api: 专属中 1/1`
- `fast_api: 共享中`
- `token: 无可用账号 0/2`

允许的状态语义：

- `手动暂停`
- `专属中 X/Y`
- `共享中`
- `无可用账号 X/Y`

说明：

- `Y` 表示该模式下的专属目标账号数
- `X` 表示该模式下当前实际分配到的专属账号数
- 若商品当前由共享池承担查询，则状态显示为 `共享中`
- 若既没有专属账号，当前又没有共享可用能力，则显示 `无可用账号`

### 6.4 商品行内展开编辑区

点击商品行后，在该行下方展开详细编辑区。

展开区结构：

1. 第一排：只读信息
   - 商品链接
   - 外部商品 ID
2. 第二排：查询参数
   - `min_wear`
   - `max_wear`
   - `max_price`
3. 第三排：三种模式目标分配输入
   - `new_api`
   - `fast_api`
   - `token`
4. 第四排：附加操作
   - `手动暂停`
   - `删除商品`

### 6.5 输入控件

- `min_wear / max_wear / max_price` 使用数字输入框
- 三种模式目标分配数使用步进器或数字输入框
- 行内变更先保留在当前配置草稿中
- 统一通过右下角 `保存` 落库

## 7. 新增商品流程

### 7.1 入口位置

新增商品入口位于当前配置工作台右上侧。

### 7.2 操作流程

1. 用户点击 `添加商品`
2. 工作台右侧展开新增商品面板
3. 用户粘贴商品 URL
4. 系统自动解析 URL 并拉取商品详情
5. 自动回填：
   - 商品名
   - 默认磨损
   - 价格参考
6. 用户补充或修改：
   - `min_wear`
   - `max_wear`
   - `max_price`
   - `new_api / fast_api / token` 目标分配数
7. 用户确认后将该商品加入当前配置草稿

### 7.3 默认值

- `min_wear` 默认沿旧逻辑带出，但允许用户修改
- `max_wear` 继续表示用户接受阈值
- 三种模式目标分配数默认全部为 `0`

## 8. 分配输入与保存校验

### 8.1 动态剩余额度提示

三种模式的输入区域都需要显示“还可分配 X”。

这里展示的不是模式总可用账号数，而是当前商品在现有草稿下还能再分到的剩余额度。

计算方式：

`剩余额度 = 当前模式总可用账号数 - 其他商品已占用的专属目标数`

注意：

- 计算时不能把当前商品自己的已有值重复扣减
- 手动暂停的商品不应继续占用专属目标额度

### 8.2 动态更新来源

“还可分配 X” 至少受两类变化影响：

1. 当前配置草稿中，其他商品分配数被修改
2. 当前账号能力池变化，例如账号新增、失效、恢复

### 8.3 保存校验

保存时必须进行模式级校验：

- `new_api` 下所有未暂停商品的专属目标分配总和，不能大于当前 `new_api` 可用账号数
- `fast_api` 同理
- `token` 同理

若超过：

- 对应输入框立即报错
- 当前配置禁止保存
- 底部操作条显示 `校验失败，无法保存`

## 9. 数据模型设计

当前的数据模型无法表达商品级模式分配与手动暂停，需要扩展。

### 9.1 QueryItem 保留的字段

继续保留：

- `query_item_id`
- `config_id`
- `product_url`
- `external_item_id`
- `item_name`
- `market_hash_name`
- `min_wear`
- `detail_max_wear`
- `max_wear`
- `max_price`
- `last_market_price`
- `last_detail_sync_at`

### 9.2 新增商品级配置字段

建议新增：

- `manual_paused: bool`

语义：

- 为 `True` 时，商品在所有模式下都不参与查询
- 被手动暂停的商品不消耗专属目标额度
- 商品状态优先显示 `手动暂停`

### 9.3 新增商品模式分配模型

建议新增独立模型，例如：

`QueryItemModeAllocation`

字段建议：

- `allocation_id`
- `query_item_id`
- `mode_type`
- `target_dedicated_count`
- `created_at`
- `updated_at`

语义：

- 表达“该商品在该模式下希望拥有多少个专属账号”
- `target_dedicated_count = 0` 不表示停用，而表示该商品默认属于该模式共享池候选

### 9.4 运行时状态模型

运行时需要单独维护每个商品每个模式的实际状态，不能直接覆盖持久化配置。

建议增加运行时快照结构，至少包括：

- `query_item_id`
- `mode_type`
- `target_dedicated_count`
- `actual_dedicated_count`
- `in_shared_pool`
- `manual_paused`
- `status`
- `status_message`

这些数据属于运行态，不写回配置真相。

## 10. 查询分配器设计

### 10.1 总体思路

继续沿用旧 `autobuy.py` 的核心思路：

- 账号 / 查询组冷却结束后上报 ready
- 分配器接收 ready
- 给 ready 的查询组派发商品

但在新后端中补齐两个旧语义之外的新层次：

1. 商品级专属目标账号数
2. 专属池 / 共享池双轨分配

### 10.2 模式边界

分配仍然按模式独立进行：

- `new_api`
- `fast_api`
- `token`

每个模式拥有自己的：

- 可用账号集合
- 专属绑定关系
- 共享池账号集合
- 共享商品轮转状态

### 10.3 专属池语义

若某商品在某模式下的 `target_dedicated_count > 0`，则该模式运行时会尽量满足它的专属目标。

规则：

- 被绑定为专属的账号 ready 后，只查询自己的绑定商品
- 只要某商品当前还有至少 `1` 个专属账号，就不进入共享池
- 专属绑定应尽量保持稳定，不要在运行时频繁抖动

### 10.4 共享池语义

以下商品属于某模式的共享池候选：

1. `target_dedicated_count = 0` 的商品
2. 原本有专属目标，但当前专属实际数已经降到 `0` 的商品

共享池规则：

- 共享账号 ready 后，按该模式的商品指针轮转逻辑分配共享商品
- 共享池不绑定固定账号
- 商品从共享池获得查询能力时，状态显示为 `共享中`

### 10.5 降级规则

当账号失效时：

1. 先移除该账号与商品的专属绑定
2. 若该商品仍保有至少 `1` 个专属账号：
   - 继续只由剩余专属账号查询
   - 不进入共享池
3. 若该商品专属账号数降到 `0`：
   - 进入共享池候选集
   - 等待共享池账号轮转查询

### 10.6 恢复规则

恢复规则不依赖“原有专属账号恢复”，而依赖“当前全局是否重新满足目标数”。

规则：

- 只要系统当前有多出来的账号
- 且已经足够重新满足某商品在某模式下的 `target_dedicated_count`
- 该商品就应立刻退出共享池
- 回到“只吃专属账号”的专属态

### 10.7 手动暂停优先级

若商品被手动暂停：

- 不参与专属分配
- 不进入共享池
- 不消耗专属目标额度
- 状态固定显示为 `手动暂停`

### 10.8 分配器稳定性原则

运行时分配器应遵守：

- 账号 ready 驱动，而不是全量忙轮询
- 既有专属绑定尽量稳定
- 只有在账号失效或资源重新满足目标时才做必要调整
- 不因一次普通查询完成就重建整个绑定关系

## 11. 商品最小可查询时间

为优化“一个商品被多个账号同时查询”的均匀度，调整最小可查询时间计算方式。

规则：

- 某商品在某模式下当前若被多个账号查询
- 其最小可查询时间改为：
  - `0.5 / 当前实际分配账号数量`

说明：

- 这里的“当前实际分配账号数量”指该模式下当前真正承担该商品查询的账号数
- 若当前实际分配账号数为 `0`，则该商品在该模式下不可调度

本次设计只锁定公式，不在设计阶段附加额外平滑因子。

## 12. 接口与前端状态需求

### 12.1 配置读写接口

查询配置接口需要补充返回：

- `manual_paused`
- 每个商品在每个模式下的 `target_dedicated_count`

### 12.2 运行时状态接口

查询运行时状态需要补充返回商品级模式状态，至少让前端能展示：

- `target_dedicated_count`
- `actual_dedicated_count`
- `status`
- `status_message`

### 12.3 前端本地草稿状态

前端需要维护当前配置草稿，用于承载：

- 未保存的参数变更
- 动态剩余额度计算
- 保存前校验结果

前端草稿状态不等于后端真实运行态。

## 13. 非目标

本次设计不做：

- 多配置并行运行
- 绑定用户手选的具体账号
- 把购买系统迁进同一页面
- 重写 legacy 查询执行器协议
- 设计最终视觉样式细节，例如颜色、动画、阴影体系

## 14. 测试重点

后续实现必须锁定以下行为：

1. 查询页支持多个配置保存与切换
2. 同一时刻只能启动一个配置
3. 新增商品时能根据 URL 自动回填详情
4. 商品行正确显示价格、磨损、目标分配数和模式状态
5. 动态“还可分配 X”计算正确
6. 保存时超配会被阻止
7. `target_dedicated_count = 0` 的商品会进入共享池候选
8. 有专属账号的商品不会吃共享池
9. 专属账号清零后会降级进共享池
10. 当全局重新满足目标数时，商品会立刻退出共享池回到专属态
11. 手动暂停商品不会参与专属和共享分配
12. `0.5 / 当前实际分配账号数量` 的最小可查询时间不回归

## 15. 结论

这次设计的核心不是把旧查询页搬到新 UI，而是补齐一套新的“查询工作台 + 商品级分配器”语义：

- UI 上：
  - 左边选配置
  - 右边改配置
  - 商品一眼看总览，按需展开编辑
- 数据上：
  - 商品保留价格与磨损配置
  - 新增商品级手动暂停与模式分配目标
- 运行时上：
  - 保留 ready 驱动调度
  - 增加专属池 / 共享池双轨模型
  - 正确处理账号失效后的降级与资源恢复后的回升
