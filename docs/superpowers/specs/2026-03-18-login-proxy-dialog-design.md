# 登录前代理确认弹窗设计

日期：2026-03-18

## 1. 目标

解决账号中心当前“发起登录”前无法直接输入或调整代理的问题。

本次改动的目标是：

- 用户点击“发起登录”后，可以先看到一个轻量代理设置弹窗。
- 弹窗默认带出当前账号已保存的代理配置。
- 用户确认后，如代理有变化，先保存代理，再继续发起登录。
- 用户取消后，不修改账号，不启动登录任务。

## 2. 当前问题

当前账号中心的登录链路存在以下交互缺陷：

1. 详情页的“代理”字段是只读展示，无法直接编辑。
2. “发起登录”按钮会立即调用后端登录接口，不给用户任何代理确认或修改机会。
3. 后端登录任务只读取账号当前持久化的 `proxy_url`，因此登录时使用哪个代理，完全取决于用户是否提前进入“编辑账号”修改并保存。

这导致实际使用流程变成：

`查看详情 -> 编辑账号 -> 修改代理 -> 保存 -> 发起登录`

这个流程虽然技术上可行，但对于“登录前临时确认代理”这个核心动作来说过于绕，且不符合用户直觉。

## 3. 范围与非目标

### 3.1 本次范围

- 为“发起登录”增加登录前代理确认弹窗。
- 弹窗支持 `direct` / `custom` 两种代理模式。
- 弹窗继续支持“完整代理 URL”和“拆分代理字段”两种录入方式。
- 发起登录前，按弹窗内容决定是否先更新账号代理配置。
- 为前端控制器和对话框补充对应测试。

### 3.2 非目标

- 不修改后端登录接口结构。
- 不增加“仅本次登录使用临时代理但不落库”的能力。
- 不把详情页改造成可编辑表单。
- 不替换现有“编辑账号”弹窗。
- 不调整 Selenium 登录底层代理实现。

## 4. 约束

本次改动必须遵守以下约束：

- 后端登录任务仍然只使用账号当前保存的 `proxy_url`。
- 用户点击确认后的行为必须稳定且可预期：
  - 代理变化时：先更新账号，再发起登录
  - 代理未变化时：直接发起登录
- 用户取消弹窗后，界面状态保持不变，不产生任何副作用。
- 复用现有代理组装规则，避免新旧两个代理拼装逻辑出现偏差。

## 5. 方案比较

### 方案 A：登录前弹轻量代理确认框

- 点击“发起登录”先弹出一个专用对话框
- 对话框只负责代理确认
- 用户确认后继续登录

优点：

- 交互最贴近用户当前诉求
- 保持详情页仍然是只读展示态
- 改动集中在前端，风险可控
- 能复用现有代理输入与组装逻辑

缺点：

- 会新增一个登录前确认步骤

### 方案 B：把详情页代理字段改为可编辑

- 详情页直接允许修改代理
- 用户改完后点击“发起登录”

优点：

- 少一个弹窗

缺点：

- 会把详情页从展示态改成半编辑态
- 需要重新处理“是否自动保存”“哪些字段可编辑”等问题
- 容易让“查看详情”和“编辑账号”职责混乱

### 方案 C：发起登录时支持传临时代理，不落库

- 前端在登录请求里直接带代理
- 后端登录任务优先使用临时代理

优点：

- 不必先更新账号

缺点：

- 需要改后端 API 合同
- 代理来源变成“账号保存值”和“临时请求值”双轨，后续维护更乱
- 不符合当前系统“登录任务只读账号状态”的建模

### 结论

采用方案 A。

它在不改变后端接口和详情页职责的前提下，以最小改动解决登录前代理无法输入的问题。

## 6. 交互设计

### 6.1 入口

用户在账号详情区域点击“发起登录”。

### 6.2 弹窗内容

弹窗名称建议为“登录代理设置”。

弹窗包含：

- 代理模式选择：`direct` / `custom`
- 完整代理输入框
- 拆分代理输入区：
  - 协议
  - 主机
  - 端口
  - 用户名
  - 密码

弹窗初始化时应根据当前账号的 `proxy_mode` 与 `proxy_url` 预填。

### 6.3 确认逻辑

用户点击确认后：

1. 根据当前输入构造 `proxy_mode` 与 `proxy_url`
2. 与当前详情账号保存值比较
3. 若代理配置有变化：
   - 调用更新账号接口，仅更新代理相关字段
   - 更新成功后再发起登录
4. 若代理配置无变化：
   - 直接发起登录

### 6.4 取消逻辑

用户点击取消后：

- 不调用更新账号接口
- 不调用登录接口
- 状态栏不应误报“正在发起登录”

## 7. 数据流

### 7.1 前端窗口层

`AccountCenterWindow` 负责：

- 创建并弹出“登录代理设置”对话框
- 在用户确认后把代理 payload 交给 controller

### 7.2 控制器层

`AccountCenterController` 负责：

- 接收 `account_id` 对应账号当前代理配置与弹窗提交配置
- 判断是否需要先更新账号
- 编排“更新代理 -> 登录”或“直接登录”两个分支
- 继续沿用现有登录任务流、冲突处理流和状态发布逻辑

### 7.3 后端层

后端无需新增字段或修改接口。

登录任务继续从账号仓储读取 `proxy_url` 并传给 `login_adapter`。

## 8. 组件拆分

建议新增一个轻量对话框，例如：

- `app_frontend/app/dialogs/login_proxy_dialog.py`

职责：

- 仅处理登录前代理确认
- 尽量复用 `_AccountDialogBase` 中已有的代理 UI 和代理组装逻辑，而不是再复制一套

如果现有代理 UI 复用成本偏高，可以把代理字段和构造逻辑进一步下沉为可复用小部件或辅助基类，但只在本次改动真正需要时才拆，不做过度重构。

## 9. 错误处理

- 更新代理失败：
  - 不继续发起登录
  - 保持现有错误展示方式
- 登录启动失败：
  - 沿用现有错误展示方式
- 对话框输入为空且模式为 `direct`：
  - 视为直连
- 对话框模式为 `custom` 但未填写任何代理：
  - 继续沿用当前代理构造规则，最终回落为 `direct`

## 10. 测试策略

### 10.1 对话框测试

补充或扩展前端对话框测试，覆盖：

- 预填当前账号代理
- `direct` 模式下构造 `None`
- `custom` 模式下完整代理输入
- `custom` 模式下拆分字段组装代理

### 10.2 控制器测试

补充控制器测试，覆盖：

- 代理变化时，先更新账号，再调用登录
- 代理未变化时，不更新账号，直接登录
- 用户取消时，不更新账号，不登录

### 10.3 回归验证

确保以下现有行为不变：

- 登录任务状态推送
- 登录冲突处理
- 登录成功后刷新账号列表
- 现有“编辑账号”弹窗仍可独立修改代理

## 11. 风险

主要风险有两个：

1. 如果登录前弹窗复制了另一套代理逻辑，后续容易与“编辑账号”弹窗行为不一致。
2. 如果控制器把“更新代理”和“发起登录”串联得过于松散，可能出现更新失败后仍然继续登录的错误行为。

因此本次实现应优先保证：

- 代理组装逻辑单一来源
- “更新成功后再登录”的顺序明确可测

## 12. 实施结论

本次采用“登录前代理确认弹窗”方案。

实现上以前端为主：

- 新增轻量登录代理对话框
- 调整账号中心窗口与控制器的登录前流程
- 复用现有代理 payload 构造逻辑
- 补齐对话框与控制器测试

后端登录接口与登录任务保持不变。
