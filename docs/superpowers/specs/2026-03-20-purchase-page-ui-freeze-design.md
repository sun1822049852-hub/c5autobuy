# 购买页 UI 冻结稿（localhost:60349）

日期：2026-03-20

## 3. 当前页面现状

当前 `app_desktop_web/src/features/purchase-system/purchase_system_page.jsx` 仍带有以下旧布局痕迹：

1. 顶部还有 Hero 大标题区。
2. 页面中部仍常驻：
   - `最近事件`
   - `账号监控`
3. 商品区还是 `expand/collapse` 心智，而不是“平铺监控行 + 按需打开抽屉”。
4. 顶部摘要里仍显示：
   - `队列中`
   - `运行代号`
   - `真实命中 实时`
   - `购买成功 已购`
   - `购买失败 回执`
5. 底部悬浮动作条目前只承担：
   - `选择配置 / 切换配置`
   - `开始扫货 / 停止扫货`

这些都不再是最终目标态。

## 4. 冻结后的产品心智

购买页不是配置管理页。

购买页是“运行中的扫货控制台”。

它的核心职责只有四件事：

1. 看当前到底在跑哪一个配置。
2. 看这个配置里的商品现在跑成什么样。
3. 在运行现场直接调整查询分配与查看命中来源。
4. 启动、停止、切换配置，并查看辅助诊断信息。

因此页面主次必须固定为：

1. 当前配置
2. 当前配置下的商品运行表现
3. 辅助信息按需弹出

而不是：

1. 欢迎头图
2. 大块摘要卡
3. 侧栏事件
4. 侧栏账号监控
5. 最后才看到商品

## 5. 页面骨架冻结

购买页自上而下固定为三层：

1. 顶部紧凑运行栏
2. 商品主列表
3. 底部悬浮动作条

### 5.1 顶部紧凑运行栏

运行栏左到右固定为：

- 当前配置名
- 当前运行状态文案
- `选择配置 / 切换配置`
- `累计购买`

运行栏明确移除：

- 大标题 Hero
- 页面描述副标题
- `队列中`
- `运行代号 / 购买代号`
- 常驻后端状态横幅

运行栏不承担诊断明细，只承担“当前是谁、当前什么状态、接下来怎么切换”的最小信息。

### 5.2 商品主列表

购买页主区直接显示当前配置中的商品列表。

每个商品默认平铺显示，不再把主指标藏进 accordion 里。

每个商品行默认可见信息固定为两组：

- 左侧：
  - 商品名称
  - 当前设置的价格
  - 当前设置的磨损
- 右侧：
  - `成功`
  - `命中`
  - `失败`
  - `查询次数`

补充约束：

- 不显示购买代号
- 不显示运行代号
- 不在主行展示队列统计
- 商品主行默认就要能完成人工扫视，不依赖展开后才能看关键指标

### 5.3 底部悬浮动作条

底部保留三个并列按钮：

- `最近事件`
- `查看账号详情`
- `开始扫货 / 停止扫货`

这里明确移除底部的 `选择配置`。

配置选择入口只保留在顶部运行栏中。

## 6. 顶部运行栏冻结细节

### 6.1 当前配置

顶部左侧显示当前配置名。

若尚未选择配置，显示：

- `未选择配置`

且此时：

- `开始扫货` 禁用
- 顶部动作文案为 `选择配置`

### 6.2 选择配置 / 切换配置

未运行时文案为：

- `选择配置`

运行中时文案为：

- `切换配置`

点击后打开居中 modal，而不是底部抽屉或侧边栏。

### 6.3 累计购买

`累计购买` 放在 `选择配置 / 切换配置` 的右侧，同属一条运行栏。

它表达的是当前运行配置上下文中的累计购买结果，不是全历史报表。

## 7. 商品主行冻结细节

### 7.1 左侧显示内容

每个商品主行左侧固定显示：

- `item_name`
- `max_price`
- `detail_min_wear ~ detail_max_wear`

说明：

- 这里展示的是配置中当前生效的磨损与价格，不是数据库里原始解析参考值。
- 展示文案应尽量贴近人眼扫视，例如：
  - `价格 <= 1234.56`
  - `磨损 0.01 ~ 0.07`

### 7.2 右侧显示内容

每个商品主行右侧固定显示四个统计位：

- `成功`
- `命中`
- `失败`
- `查询次数`

字段映射冻结为：

- `成功 = purchase_success_count`
- `命中 = matched_product_count`
- `失败 = purchase_failed_count`
- `查询次数 = query_execution_count`

文案冻结为：

- 不再显示 `已购`
- 不再显示 `回执`

统一改成：

- `成功`
- `失败`

## 8. 商品抽屉冻结

每个商品行允许打开自己的抽屉。

抽屉不是用来补看主指标，而是承担二级信息与管理动作。

抽屉固定承载两类内容：

1. 命中来源
2. 查询分配

### 8.1 命中来源

当商品产生命中时，抽屉中可看到“由哪些查询器提交了这些命中”。

这里展示的不是笼统“账号”，而是查询器来源。

查询器来源冻结为三种类型：

- `api查询器`
- `api高速查询器`
- `浏览器查询器`

实现语义分别对应：

- `new_api`
- `fast_api`
- `token`

展示层允许看到更细粒度的来源，例如：

- `账号 A / api查询器`
- `账号 A / api高速查询器`
- `账号 A / 浏览器查询器`

但分配心智不能退回成“按账号硬绑商品”。

### 8.2 查询分配

购买页承担商品分配的主要编辑入口。

也就是：

- 分配不再主要放在查询配置页里做
- 用户在购买运行现场直接改
- 修改结果仍然保存回查询配置

抽屉中的分配编辑本质上编辑的是：

- 各 mode 的目标专属数量

而不是真正把某个商品硬塞给某个账号。

换句话说，UI 上的“分配账号”在运行语义里应理解成：

- 分配查询者容量
- 分配对应 mode 的 worker 目标数

### 8.3 保存与热应用

抽屉中分配编辑采用手动保存。

流程固定为：

1. 用户修改分配
2. 点击 `保存分配`
3. 先保存配置
4. 若当前正在运行的就是这个配置，则立即热应用

热应用语义冻结为：

- 不打断当前正在跑的那一轮查询
- worker 当前轮结束后，再按新分配取下一件商品
- 不重建 HTTP session
- 不重建 browser session
- 不要求重新登录
- 不重启整个 query runtime

## 9. modal 冻结

### 9.1 最近事件

`最近事件` 从主页面移除，改为点击底部按钮后在屏幕中间弹出独立 modal。

要求：

- 独立打开
- 可拖动
- 可调整大小
- 运行期间记住位置和尺寸

### 9.2 查看账号详情

`查看账号详情` 同样从主页面移除，改为点击底部按钮后在屏幕中间弹出独立 modal。

要求与 `最近事件` 相同：

- 独立打开
- 可拖动
- 可调整大小
- 运行期间记住位置和尺寸

这里展示的是购买账号监控，不和最近事件混成一个弹窗。

### 9.3 两个 modal 的关系

冻结为：

- 各开各的 modal
- 不是一个页签弹窗
- 可以分别打开
- 可以分别拖动和改大小

## 10. 购买页与配置页的边界

购买页只负责：

- 选择当前运行配置
- 切换当前运行配置
- 启停扫货
- 查看运行中的商品表现
- 在运行现场调整商品分配

查询配置页只负责：

- 配置项管理
- 商品条目管理
- 基础参数编辑

因此冻结为：

- `开始扫货` 只在购买页
- `选择配置 / 切换配置` 只在购买页
- 商品分配主要编辑入口迁到购买页商品抽屉

但保存落点仍然是查询配置本身。

## 11. 购买页与运行时的边界

购买页是运行控制台，但不改以下运行真相：

- 查询 session 需要跨配置切换复用
- 查询负责查，购买负责买
- 查询命中提交到购买池时不能被慢统计阻塞
- 没有购买账号时，查询可以暂停；账号恢复时查询恢复

因此 UI 的任何细化都不能引入：

- 切配置就重登
- 改分配就重启运行时
- 打开 modal 就阻塞主视图轮询

## 12. 状态文案冻结

购买页主视图中使用以下文案：

- `成功`
- `命中`
- `失败`
- `查询次数`
- `最近事件`
- `查看账号详情`
- `开始扫货`
- `停止扫货`
- `选择配置`
- `切换配置`
- `累计购买`

明确废弃以下旧文案：

- `已购`
- `回执`
- `队列中`
- `购买代号`
- `运行代号`

## 13. 当前实现与冻结稿的差距

截至本文写下时，当前 web 购买页与冻结稿的主要差距是：

1. 还保留 Hero 头部。
2. 最近事件和账号监控仍常驻页面。
3. 底部动作条里还带有配置选择职责。
4. 商品区还没完全切到“主行平铺 + 抽屉承载二级内容”的最终态。
5. 商品抽屉里的“命中来源 + 查询分配热应用”闭环还没有全部到位。

## 14. 后续实现顺序建议

按风险从低到高，推荐顺序为：

1. 先把页面骨架调成：
   - 顶部紧凑运行栏
   - 商品主列表
   - 底部三按钮
2. 再把：
   - 最近事件
   - 查看账号详情
   改成两个独立 modal
3. 再把商品行整理成：
   - 左侧商品信息
   - 右侧四个主指标
4. 最后补商品抽屉：
   - 命中来源
   - 查询分配
   - 保存后热应用

## 15. 一句话冻结结论

`localhost:60349` 的购买页最终不是“带侧栏的管理页”，而是“顶部运行栏 + 商品监控主列表 + 底部悬浮动作 + 按需弹出诊断 modal”的运行控制台；分配入口前移到购买页商品抽屉，但配置持久化和 session 复用语义不变。
