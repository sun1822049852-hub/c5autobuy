# 查询运行时模式级独立商品调度设计

## 1. 目标

把当前查询运行时升级为更贴近你修订后源代码逻辑的执行模型：

- 一个查询配置包含多个商品
- 三种查询方式各自拥有一个独立调度器
- 三个调度器彼此独立，但结构和算法完全相同
- 每个调度器只面向自己那一种查询方式下的查询组
- 查询组仍然按“账号 x 查询方式”独立 ready、独立冷却、独立执行

本次改动保持以下边界不变：

- 仍只允许一个查询任务同时运行
- UI 只负责配置输入、状态展示、启动停止
- scanner 协议细节继续复用 legacy
- 命中结果仍可转发给购买运行时

## 2. 升级后的 legacy 逻辑理解

这里的重点不是“一个全局调度器服务全部模式”，而是：

1. `new_api` 有自己的调度器
2. `fast_api` 有自己的调度器
3. `token` 有自己的调度器

这三个调度器：

- 彼此独立
- 实现相同
- 各自维护当前配置下全部商品的独立轮转状态
- 各自维护同一商品在本模式内部的最小调度间隔

因此同一个商品：

- 在 `new_api` 中刚被查过
- 不影响 `fast_api`
- 也不影响 `token`

也就是说，商品调度是“模式内独立”，不是“跨模式共享”。

## 3. 当前重构版的问题

当前新运行时虽然已经能调用 legacy scanner，但在调度层面还有两个偏差：

### 3.1 调度器边界表达不清

现有实现里 `QueryTaskRuntime` 会给模式注入调度器，但文档和计划曾把目标表述成“配置级单一全局调度器”，这和你确认后的要求不一致。

### 3.2 执行语义需要明确锁定

需要明确以下语义，避免后续实现和测试再次走偏：

- 三种模式不能共享同一个商品调度状态
- 三种模式不能共用同一商品指针
- 三种模式不能共用同一商品冷却
- 同一模式下多个查询组才共享该模式自己的调度器

## 4. 新设计

### 4.1 QueryTaskRuntime

职责：

- 持有当前 `QueryConfig`
- 创建三个独立的 `QueryItemScheduler` 实例
- 建立 `mode_type -> scheduler` 映射
- 为每个模式创建对应 `ModeRunner`
- 聚合模式统计、查询组明细和最近事件

它不负责：

- 单账号执行细节
- scanner 协议细节
- 购买执行

### 4.2 QueryItemScheduler

保留一个调度器类，但会创建三个独立实例。

每个实例的职责：

- 持有当前配置下全部 `QueryItem`
- 维护本模式自己的商品轮转指针
- 维护本模式自己的商品最小调度间隔
- 当本模式下某个查询组 ready 时，返回一个商品和计划执行时间

行为约束：

- 一次只分配一个商品
- 同一商品的连续调度要满足最小冷却
- 冷却只在当前模式调度器内部生效
- 不跨模式共享任何商品状态

### 4.3 ModeRunner

每个 `ModeRunner` 只服务一个查询方式。

职责：

- 按模式筛选符合条件的账号
- 为本模式下每个符合条件账号创建独立 worker loop
- 复用本模式配置的时间窗口和冷却参数
- 当查询组 ready 时，向“本模式调度器”申请商品
- 聚合本模式统计和事件

它不负责：

- 管理其他模式的商品轮转
- 跨模式协调商品分配

### 4.4 AccountQueryWorker

继续负责：

- 单账号单模式执行一次查询
- 调用 legacy scanner
- 处理 `403` / `429` / `Not login` / 普通错误
- 输出统一事件

配合方式：

- worker loop 准备就绪后向本模式调度器申请商品
- 完成一次查询后，按当前模式配置重新进入独立冷却

## 5. 数据流

### 5.1 启动

1. `QueryRuntimeService.start(config_id)`
2. 创建 `QueryTaskRuntime`
3. `QueryTaskRuntime` 创建三个独立调度器：
   - `new_api_scheduler`
   - `fast_api_scheduler`
   - `token_scheduler`
4. 为 `new_api / fast_api / token` 分别创建 `ModeRunner`
5. 每个 `ModeRunner` 只拿到自己模式对应的调度器
6. 每个 `ModeRunner` 为自己模式下符合条件的账号启动独立 worker loop

### 5.2 单次执行

1. 查询组检查当前模式是否启用
2. 查询组检查当前是否在该模式时间窗口内
3. 查询组检查自己的冷却或退避是否结束
4. 查询组向“本模式调度器”申请一个商品
5. 调度器返回 `query_item + execute_at`
6. 查询组只执行这一个商品
7. 执行结束后更新统计和事件
8. 若命中则转发给购买运行时
9. 查询组重新进入自己的下一轮冷却

## 6. 隔离规则

这是本次设计的核心约束：

- 同一模式下，所有查询组共享该模式自己的调度器
- 不同模式之间，调度器实例必须不同
- 不同模式之间，商品轮转指针必须不同
- 不同模式之间，商品冷却状态必须不同
- 同一商品可被三个模式各自独立查询

换句话说：

- `new_api` 查过商品 A，不影响 `fast_api`
- `fast_api` 查过商品 A，不影响 `token`
- 只有同一种模式里的查询组，才会竞争同一个调度器

## 7. 状态模型

前端接口字段尽量保持兼容，重点修正后端行为语义。

保留以下任务级和模式级状态：

- `eligible_account_count`
- `active_account_count`
- `query_count`
- `found_count`
- `recent_events`
- `group_rows`

语义说明：

- `eligible_account_count`：当前模式下符合资格的账号数
- `active_account_count`：当前模式下仍具运行能力的查询组数
- `query_count`：当前模式实际执行的查询次数
- `found_count`：当前模式实际命中总数
- `group_rows`：按“账号 x 模式”展示当前查询组状态

本次不新增前端强耦合字段，避免 UI 为调度细节承担状态解释职责。

## 8. 错误处理

- `403` / `Not login`：只禁用当前查询组，不影响其他模式
- `429`：只让当前查询组进入退避，不影响其他查询组和其他模式
- 时间窗口关闭：当前模式下的查询组等待到下一次窗口开始
- 商品列表为空：当前模式调度器返回无可执行商品，但运行时不崩溃
- 单 worker 异常：记录事件，不拖垮整个查询任务

## 9. 测试重点

本次必须锁定以下行为：

1. `QueryTaskRuntime` 为三个模式创建三个不同的调度器实例
2. 同一模式下多个查询组共享该模式自己的调度器
3. 不同模式之间的调度器实例彼此独立
4. 同一商品可以被不同模式分别独立调度
5. 同一模式下商品仍按该模式自己的轮转和最小冷却执行
6. `403 / 429 / Not login` 行为不回归
7. 查询命中转购买运行时不回归

## 10. 非目标

本次不做：

- WebSocket / SSE 实时日志升级
- 查询结果持久化入库
- 多查询配置并发运行
- 移除 legacy scanner
- 改造前端协议以暴露调度器内部状态

## 11. 实施原则

- 只修正调度边界和运行时行为，不重写 scanner
- 保持前后端分层，不把调度逻辑塞进 UI
- 优先以测试锁定“模式级独立调度器”语义
- 不把“一个共享调度器”再次引回设计或测试
