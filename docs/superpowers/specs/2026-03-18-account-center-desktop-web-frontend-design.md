# 账号中心桌面化 Web 前端设计

日期：2026-03-18

## 1. 目标

为当前项目新增一个独立的桌面化 Web 账号中心程序，用来替代现有难用的 `PySide6` 账号中心界面，但继续复用现有 Python 后端、数据库和账号业务逻辑。

第一版目标不是迁完整个桌面程序，而是先把“账号中心”做成一个可独立启动、可实际使用、并且后续可继续承载 `查询系统 / 购买系统` 迁移的 Electron 桌面应用骨架。

## 2. 当前问题

现有账号中心虽然已经完成了四列主列表改造，但本质上仍然有几个结构性问题：

1. 前端仍是 `PySide6 Widgets`，视觉与交互上限明显低于桌面化 Web 前端。
2. 登录、代理、购买配置虽然已经能用，但在桌面控件风格下仍然显得割裂，难以形成一体化工作台。
3. 当前桌面程序的整体外壳不适合逐步迁移更多模块，后续继续搬 `查询系统 / 购买系统` 会越来越别扭。
4. 用户明确反馈当前 UI 有大问题，希望逐步迁移到类似浏览器页面但仍以桌面程序方式打开的前端形态。

## 3. 已确认的用户决策

以下内容已经在讨论中确认：

- 新程序是一个独立的新桌面程序，不嵌回现在的 `PySide6 WorkspaceWindow`。
- 技术壳选择 `Electron + Node.js`。
- 新程序启动时应自动拉起本地 Python 后端，不要求用户手动启动服务。
- 整体布局采用左侧导航式桌面应用骨架。
- 左侧导航第一版显示：
  - `账号中心`
  - `查询系统（占位）`
  - `购买系统（占位）`
- 首页骨架采用“紧凑控制台型”：
  - 页头更紧凑
  - 概览卡片和工具条压缩在上方
  - 主列表尽量早露出
- 首页主区域结构选择：
  - 主列表为主
  - 列表下方增加状态/日志带
- 编辑动作打开方式选择：
  - 小改动使用弹窗
  - 复杂流程使用右侧抽屉
- 登录范围第一版选择：
  - 迁 `发起登录 + 登录任务状态展示`
  - 暂不迁冲突处理
- 数据关系选择：
  - 共用账号业务数据
  - 新程序单独维护自己的 UI 状态缓存

## 4. 方案比较

### 方案 A：Electron + React + Python 后端自启动

- Electron 负责桌面壳、窗口、子进程生命周期管理。
- React 负责界面渲染和状态组织。
- 现有 Python `FastAPI` 继续提供账号中心和登录接口。

优点：

- 最适合后续继续迁移 `查询系统 / 购买系统`。
- 组件边界清晰，页面状态容易维护。
- 复杂交互和桌面体验更容易持续演进。

缺点：

- 第一版搭建成本高于纯 HTML/JS。

### 方案 B：Electron + 原生 HTML/CSS/JS + Python 后端自启动

- 保留 Electron 桌面壳。
- 页面直接使用原生 DOM 和脚本组织。
- 风格参考 `cs2_alchemy` 的现有桌面 Web 方案。

优点：

- 第一版起步更快。
- 与参考项目的写法最接近。

缺点：

- 页面一多后状态和交互很容易失控。
- 后续迁模块时可维护性差于 React。

### 方案 C：Electron + Node UI server + Python 业务后端

- Electron 加载 Node 提供的本地 UI server 页面。
- Python 后端继续保留业务接口。

优点：

- 与参考项目结构最像。

缺点：

- 现在项目本身已经有成熟 Python 后端，再加一层 Node UI server 会导致边界混乱。
- 容易让 Node 和 Python 同时承担后端职责。

### 结论

采用方案 A。

原因：

- 当前项目的 Python 后端已经成型。
- 新目标不是做一个一次性的账号页面，而是搭出可继续迁移其他模块的桌面 Web 壳。
- 因此最合理的结构是：
  - `Electron` 负责桌面运行壳
  - `React` 负责界面
  - `Python/FastAPI` 继续负责业务

## 5. 总体架构

### 5.1 应用边界

第一版新增一个独立桌面程序：

- 与当前 `run_app.py -> PySide6` 主程序并存
- 只承载账号中心页面
- 但外壳层级按“未来可继续迁移模块”的方式搭建

整体结构如下：

1. Electron Main Process
   - 创建桌面窗口
   - 管理 Python 后端子进程
   - 提供最小桌面桥接能力

2. Electron Renderer
   - React 页面
   - 左侧导航
   - 账号中心首页
   - 弹窗与抽屉

3. Python Backend
   - 继续使用当前 `FastAPI`
   - 继续使用现有 service / repository / SQLite
   - 继续负责登录、账号编辑、购买配置、任务状态查询

### 5.2 新旧程序关系

- 旧 `PySide6` 程序继续保留，短期内不删。
- 新 Electron 账号中心通过 HTTP 调现有 Python 接口。
- 两者共用同一份业务数据，不各自维护独立账号真相。

## 6. 桌面壳设计

### 6.1 运行方式

用户双击新的 Electron 程序后：

1. Electron 主进程启动。
2. 检查 Python 运行环境与项目工作目录。
3. 拉起本地 Python 后端子进程。
4. 等待后端健康检查通过。
5. 创建桌面窗口并加载 React 页面。

### 6.2 左侧导航

左侧导航第一版固定显示：

- `账号中心`
- `查询系统（即将迁移）`
- `购买系统（即将迁移）`

其中：

- `账号中心` 可进入真实页面
- 其余两个先是占位态，不实现业务功能

这层导航不是装饰，而是提前把未来整体桌面 Web 程序的骨架搭好。

### 6.3 风格方向

视觉风格参考 `cs2_alchemy` 里 `Electron BrowserWindow + 左侧导航 + 主工作区` 的结构语言，但不照搬其 Node UI server 模式。

风格要求：

- 偏桌面控制台，不做营销站式大卡片堆砌
- 页面层次清晰，列表优先
- 颜色和密度统一，不再出现旧 `PySide6` 界面的割裂感

## 7. 账号中心首页结构

### 7.1 总体组织

首页采用“紧凑控制台型”布局：

1. 顶部页头
2. 概览卡片
3. 主列表工具区
4. 主账号列表
5. 列表下方状态/日志带

核心原则：

- 列表是主操作区
- 状态带只负责反馈，不重新变成旧版详情面板

### 7.2 顶部页头

页头左侧：

- 页面标题，例如 `C5 账号中心`
- 一句简短副标题，例如 `统一管理账号、登录与购买配置`

页头右侧：

- `刷新`
- `添加账号`

不在页头堆砌更多次要按钮。

### 7.3 概览卡片

第一版固定四张：

- `总账号`
- `未登录`
- `无 API Key`
- `可购买`

建议行为：

- 卡片支持点击
- 点击后直接对主列表施加对应筛选

这样可以让用户快速聚焦待处理账号。

### 7.4 主列表

列表固定四列：

- `C5昵称`
- `API Key`
- `购买状态`
- `代理`

列表行为沿用已确认的业务规则：

- `C5昵称`：编辑备注
- `API Key`：编辑或清空 key
- `代理`：修改代理
- `购买状态`：
  - 若为未登录，触发登录抽屉
  - 否则进入购买配置抽屉

删除账号仍使用右键菜单，不改成常显按钮。

### 7.5 主列表工具区

工具区保持轻量：

- 搜索框
- 基础筛选入口
- 必要时保留刷新入口

不在第一版加入复杂的高级筛选系统。

### 7.6 列表下方状态/日志带

状态/日志带不是账号详情区，而是反馈区。

第一版建议包含三块：

- 最近登录任务
- 最近错误
- 最近修改

作用：

- 承接“刚才点登录之后发生了什么”
- 承接“哪个账号修改失败了”
- 承接“刚才哪一项配置被改动了”

## 8. 编辑与配置交互

### 8.1 弹窗

以下轻量编辑使用居中弹窗：

- 添加账号
- 修改备注
- 修改 API Key
- 修改代理

原则：

- 快进快出
- 输入项少
- 操作完成后直接刷新列表或局部更新

### 8.2 右侧抽屉

以下复杂流程使用右侧抽屉：

- 登录流程
- 购买配置

这样可以承接更长的状态链路，而不必塞进小弹窗。

### 8.3 登录范围

第一版登录范围只到：

- 发起登录
- 展示登录任务状态

明确不包含：

- 登录冲突处理

冲突处理留到后续阶段，避免第一版范围失控。

### 8.4 代理修改后的行为

代理修改属于小编辑弹窗，但保存行为要与登录流程联动：

- 代理未变化：只保存/关闭
- 代理有变化：保存后自动切到登录抽屉，并展示登录任务状态

## 9. 数据流与接口边界

### 9.1 数据真相来源

新前端不直接读 SQLite，不直接触碰 Python 内部对象。

统一边界只有：

- Electron Renderer
- HTTP 调用 Python `FastAPI`

### 9.2 共用数据

新旧程序共用的数据包括：

- 账号基础信息
- 登录状态
- 备注
- API Key
- 代理
- 购买配置
- 当前仓库
- 登录任务状态

### 9.3 新程序独立的 UI 状态

新程序单独保存：

- 当前导航页
- 搜索关键词
- 过滤条件
- 卡片筛选状态
- 表格列宽
- 状态带展开收起状态
- 窗口大小和位置

即：

- 业务数据共享
- UI 状态隔离

### 9.4 第一版依赖的后端接口

直接复用现有能力：

- `GET /account-center/accounts`
- `PATCH /accounts/{account_id}`
- `PATCH /accounts/{account_id}/purchase-config`
- `GET /purchase-runtime/accounts/{account_id}/inventory`
- `POST /accounts/{account_id}/login`
- `GET /tasks/{task_id}` 以及对应任务状态读取能力

必要时允许为 Electron 启动补充：

- 健康检查接口
- 更明确的启动就绪信号

但不重写现有业务逻辑。

## 10. 前端实现分层

推荐的 React 结构：

1. `Shell`
   - 左侧导航
   - 页头
   - 应用级状态

2. `AccountCenterPage`
   - 概览卡片
   - 搜索/筛选
   - 主列表
   - 状态/日志带

3. `AccountTable`
   - 四列渲染
   - 行点击
   - 右键菜单

4. `Dialogs`
   - 添加账号
   - 备注编辑
   - API Key 编辑
   - 代理编辑

5. `Drawers`
   - 登录抽屉
   - 购买配置抽屉

6. `client`
   - 统一封装所有后端 HTTP 调用

7. `desktop bridge`
   - Electron preload 暴露的桌面能力
   - 比如窗口状态、后端启动状态、进程退出通知

## 11. 错误处理

### 11.1 启动阶段

- Python 后端启动失败：显示明确的启动失败页，不允许白屏
- 健康检查超时：显示重试入口和错误摘要

### 11.2 页面加载

- 列表加载失败：列表区域显示错误态
- 不用多个随机弹窗去打断

### 11.3 编辑与流程

- 单项编辑失败：使用页内提示 + 状态带记录
- 登录启动失败：在登录抽屉顶部展示错误
- 购买配置保存失败：抽屉内原地报错，不自动关闭抽屉

原则：

- 错误优先留在当前上下文中处理
- 减少全局弹窗打断

### 12.1 本次必须做

- Electron 独立桌面程序壳
- Python 后端自动启动
- 左侧导航骨架
- 账号中心首页
- 概览卡片
- 主列表四列
- 搜索/基础筛选
- 添加账号
- 修改备注
- 修改 API Key
- 修改代理
- 购买配置抽屉
- 发起登录
- 登录任务状态展示
- 列表下方状态/日志带

- 查询系统迁移

- 复杂皮肤/主题系统

## 13. 实施建议

建议分三步推进：

### Step 1：跑通新桌面壳

- Electron 主进程
- Python 子进程启动
- 健康检查
- React Shell
- 左侧导航

### Step 2：迁账号中心主界面

- 页头
- 概览卡片
- 主列表
- 搜索与筛选
- 基础编辑弹窗

### Step 3：补复杂流程

- 登录抽屉
- 登录任务状态展示
- 购买配置抽屉
- 列表下方状态/日志带

## 14. 验收标准

第一版可交付标准：

1. 双击新程序可直接打开账号中心。
2. 无需手动启动 Python 后端。
3. 左侧导航可见，账号中心可正常进入。
4. 主列表四列能正确加载。
5. 概览卡片能反映真实数据。
6. 可完成备注、API Key、代理编辑。
7. 可进入购买配置并保存。
8. 可发起登录并看到任务状态。
9. 整体交互明显优于当前 `PySide6` 账号中心。

## 15. 结论

本次采用：

- `Electron + React` 桌面化 Web 前端
- `Python/FastAPI` 继续作为唯一业务后端
- 独立新程序承载账号中心
- 左侧导航预留未来模块迁移入口
- 第一版先把账号中心做透，再继续逐步迁移其他部分

这套设计既满足用户对“更美观、更流畅、桌面化 Web 前端”的要求，也避免重写 Python 底层业务逻辑，并为后续迁移 `查询系统 / 购买系统` 提前打好外壳和边界。
