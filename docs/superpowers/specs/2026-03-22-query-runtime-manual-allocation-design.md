# 查询运行时手动分配设计

日期：2026-03-22

## 1. 文档目的

本文用于冻结“配置管理中的查询账号期望分配”与“购买页中的运行时实际分配”两套语义，避免后续实现继续把配置层与运行时层混在一起。

本文只服务当前新 UI 与新后端，不为旧 UI、兼容层或 legacy 入口保留额外约束。

若与以下旧文档局部冲突，以本文为准：

- `docs/superpowers/specs/2026-03-20-purchase-runtime-drawer-allocation-design.md`
- `docs/superpowers/specs/2026-03-21-config-management-density-refresh-design.md`

## 2. 范围

本文覆盖：

- 配置管理页“期望分配”的语义边界
- 购买页“实际分配”的运行时语义
- 查询 runtime 的内存态实际分配模型
- 开始扫货时的初始分配规则
- 运行中手动调整、释放、暂停、删除、失效、恢复的行为
- 共享池的商品候选规则
- 购买页的交互模型、草稿态、提交与离开保护
- 后端 runtime API 与状态读模型边界

本文不覆盖：

- 旧 UI / `pyui` 的适配
- 购买执行链路的重写
- 查询器监控弹窗的最终视觉稿
- 跨进程、跨重启持久化运行时实际分配

## 3. 当前问题

结合当前代码状态，存在以下问题：

1. 配置管理页保存的 `mode_allocations.target_dedicated_count` 与购买页当前展示的“分配”共用了同一套字段，但两者其实不是同一层语义。
2. 当前购买页修改分配时，本质仍在改配置并调用热应用，而不是修改 runtime 中的实际查询器归属。
3. 当前查询 allocator 会持续参考配置目标值进行自动补位，这与“运行中由用户接管实际分配”的需求冲突。
4. 共享池与专属分配的边界尚未正式文档化，容易在重构中被 AI 或后续实现误写成“自动回填期望值”。
5. 购买页缺少独立的草稿态与统一提交动作，无法表达“实际分配编辑但尚未提交”的状态。

## 4. 已确认的用户决策

以下内容已经在会话中明确拍板：

- 配置管理中的查询账号分配，表示“期望得到的账号数量”，不是运行时强制绑定。
- 购买页中的账号分配，表示“运行中的实际分配”，不等同于配置。
- 开始扫货时，系统根据配置中的期望值，对当前配置做一次初始分配。
- 初始分配完成后，运行中的实际分配状态只允许由用户手动调整，不再按配置期望自动重平衡。
- 同一配置持续使用时，保留当前运行时实际分配状态。
- 同一配置 stop / start 时，保留当前运行时实际分配状态。
- 切换到其他配置时，当前配置的运行时实际分配状态清空。
- 之后再切回旧配置时，从配置期望值重新做初始分配，不恢复旧手调结果。
- 商品被暂停、删除时，该商品上被释放的查询器直接进入共享池。
- 查询器失效时，从当前实际分配中移除。
- 查询器恢复时，直接进入共享池，不回原商品。
- 共享池只服务“当前在该 mode 下一个专属查询器都没有”的商品。
- 只要某商品在某个 mode 下还有哪怕 1 个专属查询器，它就不进入共享池。
- 如果所有查询器都已被分配为专属，则共享池为空；此时没有专属查询器的商品在该 mode 下就不会被查询。
- 初始分配资源不足时，优先“铺开覆盖”，尽量先让更多商品各拿到 1 个查询器，而不是先满足排在前面的商品。
- 购买页 UI 显示“实际分配 / 配置期望”。
- 购买页中的 `+ / -` 只表示数量变化；用户不需要感知被分配或被释放的是哪一个具体查询器。
- 购买页移除自动防抖提交，改为页面底部统一 `提交更改` 按钮。
- 购买页存在未提交草稿时，切换配置或离开页面需要弹出 `保存 / 不保存` 提示。

## 5. 术语定义

### 5.1 期望分配

配置管理页中保存的数量目标，表示：

- 在开始扫货时，该商品在该 mode 下希望拿到多少个专属查询器。

它只用于“初始分配”，不用于运行中的自动补位。

### 5.2 实际分配

查询 runtime 当前内存中的真实查询器归属，表示：

- 当前哪些具体查询器在查哪个商品。

实际分配只受以下行为影响：

- 初始分配
- 用户手动增减
- 商品暂停 / 删除
- 查询器失效 / 恢复

### 5.3 共享池

某个 mode 下暂未专属绑定、且当前可用的查询器集合。

共享池只轮转查询：

- 当前在该 mode 下 `actual_assigned_count = 0`
- 且商品未暂停、未删除

的商品。

## 6. 方案比较

### 方案 A：购买页继续直接修改配置目标值

优点：

- 改动最少

缺点：

- 配置层与运行时层继续混淆
- 用户以为自己在改运行时，实际改的是配置
- 运行中会持续受配置目标值牵引，无法满足“用户接管后不自动回补”的要求

### 方案 B：购买页维护 runtime-only 的实际分配，配置目标值只参与初始分配

优点：

- 配置意图与运行时状态职责清晰
- 完全符合当前用户需求
- 能自然表达共享池、释放、恢复、离开保护等行为

缺点：

- 需要新增 runtime 分配状态与接口
- 需要改造当前 allocator 逻辑

### 方案 C：购买页实际分配也落数据库持久化

优点：

- 刷新和重启后还能恢复运行时实际分配

缺点：

- 会把 runtime 状态伪装成配置
- 与“切换配置即清空运行时分配”的需求冲突
- 会增加状态一致性复杂度

## 7. 推荐方案

采用方案 B：

- 配置管理页继续保存期望分配
- 开始扫货时按期望值做一次初始分配
- 运行时维护一份 runtime-only 的实际分配表
- 运行中实际分配不再按期望值自动重平衡
- 被释放或恢复的查询器统一进入共享池

## 8. 运行时状态模型

### 8.1 查询器标识

运行时操作对象虽然在 UI 中以“数量”出现，但后端仍以具体查询器为最小单位。

建议查询器稳定标识为：

- `account_id:new_api`
- `account_id:fast_api`
- `account_id:token`

### 8.2 runtime 分配状态

建议在查询 runtime 会话内按 `mode` 挂载内存态分配表：

- `queryers_by_id`
- `item_assignments`
- `shared_queryers`
- `shared_items`
- `shared_pointer`

建议字段如下：

#### `queryers_by_id`

每个查询器一条记录：

- `queryer_id`
- `account_id`
- `account_display_name`
- `mode_type`
- `available`
- `assigned_query_item_id`
- `assignment_source`
  - `initial`
  - `manual`
  - `shared`
- `pending_assignment`

#### `item_assignments`

- `query_item_id -> set(queryer_id)`

用途：

- 快速计算某商品在某 mode 下的 `actual_assigned_count`

#### `shared_queryers`

- 当前在共享池中、且可用的查询器有序列表

#### `shared_items`

- 当前满足共享池候选条件的商品有序列表

#### `shared_pointer`

- 共享轮转指针

## 9. 初始分配规则

### 9.1 触发时机

只在以下时机触发：

- 点击“开始扫货”
- 或切换配置后重新开始该配置

### 9.2 候选范围

某个 mode 的初始分配只考虑：

- 当前该 mode 下可用的查询器
- 当前配置中未暂停、未删除的商品
- 其中 `target_dedicated_count > 0` 的商品参与专属初始分配

### 9.3 分配算法

采用“铺开优先”的轮次分配：

1. 第一轮，给所有目标值大于 0 且尚未拿到专属查询器的商品尽量各发 1 个查询器。
2. 若还有剩余查询器，再进行第二轮，继续给仍未达到目标值的商品各发 1 个。
3. 直到查询器发完，或所有目标满足。

示例：

- 目标值为 `5 / 2 / 1`
- 可用查询器只有 `4`

结果应为：

- 第一轮：`1 / 1 / 1`
- 第二轮：`2 / 1 / 1`

而不是：

- `4 / 0 / 0`

### 9.4 分配结果

- 拿到专属查询器的商品进入专属态
- 没有拿到任何专属查询器的商品，若未暂停，则进入共享候选态
- 未被专属分走的查询器进入共享池

## 10. 运行中调度规则

### 10.1 专属态

只要某商品在某 mode 下 `actual_assigned_count > 0`：

- 它就只由这些专属查询器查询
- 它不进入共享池

### 10.2 共享候选态

只要某商品在某 mode 下：

- `actual_assigned_count = 0`
- 且商品未暂停、未删除

它就进入该 mode 的共享候选列表。

### 10.3 共享池为空

如果某个 mode 下：

- 所有可用查询器都已被专属分走

则：

- `shared_queryers` 为空
- 所有 `actual = 0` 的商品在该 mode 下都不会被查询

### 10.4 运行中不自动补位

进入运行后，系统不再因为配置中的期望值不足而自动重平衡。

即：

- 不自动把共享池中的查询器重新分给“目标值还没满足”的商品
- 不自动把恢复的查询器补回旧商品

## 11. 运行中状态变更规则

### 11.1 用户手动增加实际分配

购买页某商品某 mode 点一次 `+`：

- 前端仅增加本地草稿值
- 提交时，后端从共享池中任选一个可用查询器分给该商品

结果：

- 该商品 `actual + 1`
- 若此前 `actual = 0`，则它退出共享候选态

### 11.2 用户手动减少实际分配

购买页某商品某 mode 点一次 `-`：

- 前端仅减少本地草稿值
- 提交时，后端从该商品当前已绑定的查询器中任选一个释放

结果：

- 被释放的查询器进入共享池
- 若该商品释放后 `actual = 0`，则它进入共享候选态

用户不需要知道被释放的是哪一个具体查询器。

### 11.3 商品被暂停或删除

当商品被暂停或删除时：

- 该商品上的全部专属查询器在当前任务完成后释放
- 被释放的查询器全部进入共享池

其中：

- 暂停商品不参与共享池查询
- 删除商品直接从 runtime 视图和配置中移除

### 11.4 查询器失效

查询器失效时：

- 从当前实际分配中移除
- 若该商品因此变成 `actual = 0`，则进入共享候选态

### 11.5 查询器恢复

查询器恢复时：

- 直接进入共享池
- 不回原商品

### 11.6 生效时机

所有分配变更统一遵循：

- 不打断当前查询任务
- 当前任务完成后更新分配状态
- 下一次冷却就绪时按新状态执行

## 12. UI 交互设计

### 12.1 配置管理页

继续展示并编辑：

- `配置期望`

配置管理页不负责运行时实际分配。

### 12.2 购买页展示

购买页每个商品每个 mode 显示：

- `实际分配`
- `配置期望`

其中：

- `实际分配` 可编辑
- `配置期望` 只读

### 12.3 购买页编辑方式

购买页 `实际分配` 使用 `+ / -` 做数量调整：

- 点 `+`：本地草稿 `+1`
- 点 `-`：本地草稿 `-1`

虽然 UI 是数字交互，但后端真正做的是“具体查询器”的增减。

### 12.4 页面内可用数限制

前端按 `mode` 维护共享池本地可用余额：

- 某商品点一次 `+`，同 mode 的共享池余额 `-1`
- 某商品点一次 `-`，同 mode 的共享池余额 `+1`

这样可以防止用户把本地草稿点到超过当前页面已知可用数。

### 12.5 提交方式

移除自动防抖提交。

页面底部增加全局按钮：

- `提交更改`

点击后一次性提交当前页面所有未提交的实际分配草稿。

### 12.6 离开保护

购买页存在未提交草稿时：

- 切换配置
- 切换页面
- 离开购买页

都需要弹出确认框：

- `保存`
- `不保存`

规则与当前配置管理页未保存保护保持一致。

## 13. 后端接口建议

### 13.1 写接口归属

运行时实际分配属于 query runtime，因此写接口归 `query-runtime`。

购买页仍继续通过 `purchase-runtime/status` 读取聚合后的展示状态。

### 13.2 新增接口

建议新增：

#### `POST /query-runtime/queryers/{mode_type}/{account_id}/assign`

请求：

- `config_id`
- `query_item_id`

语义：

- 把该具体查询器分配到目标商品

#### `POST /query-runtime/queryers/{mode_type}/{account_id}/release`

请求：

- `config_id`

语义：

- 把该具体查询器从当前商品释放回共享池

### 13.3 统一提交接口

由于购买页采用“本地草稿 + 提交更改”模式，首版更推荐增加批量提交接口：

#### `PUT /query-runtime/configs/{config_id}/manual-assignments`

请求内容表示当前页面提交后的运行时实际分配结果，按 mode 传递“商品应有的实际数量变化”或具体查询器增减指令。

实现上可以：

- 前端先按数量草稿生成增减操作列表
- 后端再把这些操作映射成具体查询器的 `assign / release`

### 13.4 读模型扩展

`GET /purchase-runtime/status` 需要补充：

- 每个商品每个 mode 的 `target_dedicated_count`
- `actual_dedicated_count`
- `status`
- `status_message`
- 当前共享池可用数量

`GET /query-runtime/status` 建议同步补充更细的调试字段：

- `queryer_rows`
- `shared_queryer_count`
- `shared_item_count`

## 14. 错误处理与降级

### 14.1 提交时共享池已变化

主交互路径由前端限制可用数。

后端仍需要做最小兜底：

- 若提交时共享池实际数量不足，只应用当前仍可应用的部分
- 并返回最终 runtime 结果

但这只是竞态保护，不作为主交互设计的一部分。

### 14.2 配置切换

用户切换到其他配置时：

- 若当前购买页有未提交草稿，则先弹出保存 / 不保存提示
- 切换成功后，旧配置的 runtime 实际分配清空

### 14.3 配置被删除

- 若当前购买页正在编辑该配置，先按离开保护处理
- 删除后 runtime 中对应配置的实际分配状态直接清空

## 15. 验收标准

满足以下条件即可认为本设计落地正确：

1. 配置管理页编辑的是期望分配，而不是运行时实际分配。
2. 购买页编辑的是实际分配，而不是配置。
3. 点击开始扫货时，系统按期望值对当前配置做一次初始分配。
4. 初始分配在资源不足时采用“铺开优先”。
5. 进入运行后，系统不再按期望值自动补位。
6. 只要某商品在某 mode 下 `actual > 0`，它就不进入共享池。
7. 共享池只查询 `actual = 0` 且未暂停的商品。
8. 共享池为空时，`actual = 0` 的商品在该 mode 下不被查询。
9. 商品暂停、删除、查询器失效、查询器恢复后，被释放或恢复的查询器都进入共享池。
10. 查询器恢复后不会自动回原商品。
11. 购买页 `+ / -` 只调整本地草稿，不立刻提交。
12. 页面底部 `提交更改` 会一次性提交全部 runtime 草稿。
13. 未提交草稿时，切换配置或离开购买页会弹出保存 / 不保存确认。

## 16. 后续实现顺序建议

1. 拆分当前 allocator，区分“初始分配器”和“运行时实际分配状态”
2. 在 query runtime 中挂载内存态实际分配表
3. 扩展 query runtime / purchase runtime 状态读模型
4. 增加 runtime 手动分配提交接口
5. 改造购买页，把“改数字并保存配置”改成“编辑实际分配草稿并统一提交”
6. 复用当前未保存保护模式，为购买页增加离开保护
7. 补齐前后端测试，覆盖：
   - 初始铺开分配
   - 共享池候选规则
   - 手动增减
   - 释放进共享池
   - 失效恢复进共享池
   - 未提交草稿离开保护
