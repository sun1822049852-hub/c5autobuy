# 程序会员与共享工作区冻结设计

日期：2026-04-19

## 1. 目标

本文件用于冻结当前项目对“程序会员 / 程序账号 / 本地工作区”的最终语义，覆盖此前对“本地多租户”“按会员隔离本地业务数据”“切账号是否切工作区”等问题的反复摇摆。

本次冻结后的唯一准绳是：

- 远端统一鉴权与会员判定
- 本地只有一份共享业务数据
- 程序账号只承载权限，不承载本地数据归属
- 切换程序账号只改变权限态，不切换本地工作区
- 登出或会员失效后，本地数据仍可只读查看，但关键功能必须锁死

## 2. 核心冻结结论

### 2.1 本地工作区只有一份

当前本地程序只维护一份共享工作区。

这份共享工作区包含但不限于：

- C5 账号
- 查询配置
- 购买配置
- 浏览器 session / bundle
- 本地运行配置
- 本地运行时状态
- 本地日志与诊断数据

这些数据继续只留在用户机器，不上传远端，不按程序会员账号拆库、拆目录、拆 session，也不随程序账号切换而切换。

### 2.2 程序账号不是本地数据所有者

程序账号的职责仅限于：

- 远端登录态承载
- 会员有效性判断
- 套餐 / 功能权限判断
- 设备校验
- 续期、吊销、失效等远端控制面语义

程序账号不是本地工作区的 owner，不负责决定“这份本地数据属于哪个账号”。

### 2.3 切换程序账号只改变权限态

切换程序账号的真实语义不是“切租户”，而是“换一把权限钥匙”。

因此：

- 切到无会员账号，不切换本地工作区
- 切到有会员账号，也不切换本地工作区
- 切换前后看到的仍是同一份本地共享数据
- 差别只在于当前账号是否有权使用这份数据执行关键功能

### 2.4 登出或会员失效不清本地数据

程序账号主动登出、会员过期、远端明确失效，默认都不删除本地数据。

这些情况的统一结果是：

- 本地数据继续保留
- 用户仍可查看本地数据
- 关键动作全部锁死

这不是“切换到空工作区”，也不是“清档”，而是进入只读锁定态。

## 3. 权限状态模型

当前产品语义建议冻结为以下两层：

### 3.1 工作区状态

- `shared_workspace`
  - 永远只有一份
  - 不随程序账号变化

### 3.2 程序访问状态

- `local_pass_through`
  - 当前开发 / 骨架阶段
  - 关键动作仍放行
- `unlocked`
  - 已登录远端程序账号
  - 当前会员 / 权限有效
  - 可执行关键动作
- `readonly_locked`
  - 未登录、主动登出、会员无效、会员过期、吊销、不可用套餐、设备冲突等
  - 共享工作区只读可见
  - 关键动作全部禁用

## 4. 只读锁定态语义

### 4.1 允许的动作

只读锁定态下，允许继续进行：

- 查看页面
- 查看账号列表与配置内容
- 查看查询 / 购买状态快照
- 查看日志、诊断、历史记录
- 查看当前程序会员状态
- 切换或重新登录程序账号
- 手动停止当前运行

### 4.2 禁止的动作

只读锁定态下，必须统一拦截：

- 新增、编辑、删除账号
- 发起账号登录
- 修改查询配置
- 修改购买配置
- 修改查询设置
- 修改购买 / 运行设置
- 启动查询运行时
- 启动购买运行时
- 切换并应用会影响运行态的配置
- 打开 / 同步 OpenAPI 白名单
- 任何会改写本地状态或触发外部执行的关键动作

## 5. 远端与本地职责边界

### 5.1 远端控制面负责

- 程序账号登录
- 程序账号登出
- 程序会话刷新
- 会员状态查询
- 套餐 / 功能权限判定
- 设备冲突、吊销、宽限、续期等控制语义

### 5.2 本地程序负责

- 保存共享工作区数据
- 在本地展示共享数据
- 把关键动作统一接到 Program Access guard
- 根据远端返回的权限态决定是 `unlocked` 还是 `readonly_locked`
- 在不清空共享数据的前提下切换程序访问状态

## 6. 当前仓库的方向判定

### 6.1 应保留的方向

以下方向与本次冻结结论一致，应保留：

- 本地 Program Access 抽象
- route 层统一关键动作 guard
- `/app/bootstrap` 暴露 `program_access`
- frontend 统一 Program Access 错误出口
- frontend 左侧“程序会员区”作为远端控制面入口与状态展示位
- “先进入本地界面，再在关键动作处拦截”的总体路径

### 6.2 应停止的方向

以下方向不再符合当前产品语义，应停止继续推进到本地桌面主线：

- 本地业务数据按 `owner_user_id` 隔离
- 本地账号、配置、session、运行时目录随程序账号切换
- 把程序账号当作本地数据 owner
- 通过切换程序账号切换本地工作区
- 为当前本地桌面主线继续扩展 `MultiTenant*RuntimeService`

### 6.3 可复用但应迁移位置的内容

以下能力仍有价值，但应视为“未来远端控制面能力”，而不是继续塞进本地共享业务数据层：

- `/auth/*` 契约
- `AuthGuard`
- token service
- password hasher
- membership summary / limits 契约

## 7. 对现有文档的裁决

### 7.1 `2026-04-06-membership-auth-design.md`

这份文档对“远端统一鉴权、会员在服务端判断”的大方向仍有参考价值，但以下内容对当前本地桌面语义已经过期：

- 现有本地业务表统一挂 `owner_user_id`
- 运行时目录按用户拆分
- repository 默认只读写当前用户数据
- 本地工作区按会员 / 用户做多租户隔离

因此，该文档在“当前本地桌面架构”语境下应视为部分过期。

### 7.2 `2026-04-13-local-program-access-extension-design.md`

这份文档关于以下部分仍然有效：

- 远端控制面只负责程序会员控制链
- 本地业务数据继续留在用户机器
- Program Access gateway / guard 入口
- 关键动作矩阵
- 查询 / 购买内循环不反复做会员校验

但它尚未把下面这条语义写死：

- 本地工作区只有一份共享数据
- 切换程序账号只改变权限态，不改变本地工作区
- 登出 / 会员失效后进入只读锁定态，而不是切租户或清数据

因此，该文档应视为“仍有效，但被本文件进一步冻结补完”。

## 8. 交接结论

后续若继续推进程序会员控制链，应以本文件为最新冻结边界：

- 共享工作区只有一份
- 程序账号只是权限钥匙
- 切账号不切数据
- 无会员时只读可见
- 有会员时在原工作区上直接恢复可用

任何新设计如果再次回到“按程序账号切本地工作区”或“按程序账号隔离本地业务数据”，都视为跑偏。
