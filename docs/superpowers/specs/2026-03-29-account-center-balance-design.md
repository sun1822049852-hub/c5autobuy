# 账号中心余额展示设计

日期：2026-03-29

## 1. 目标

在账号中心右侧列表中新增余额展示，并统一由后端负责余额获取、缓存和刷新调度。

本设计同时满足以下业务要求：

- 主链优先使用 OpenAPI 余额接口。
- 当账号登录完成但尚未拿到 `api_key` 时，允许使用浏览器 session 兜底。
- 余额不要求实时刷新，后续使用中改为 8-10 分钟随机时间刷新一次。
- 浏览器 session 兜底失败时不重试，避免把接口打爆。
- 浏览器链路与 API 链路必须严格区分代理。

## 2. 主链与备用链

### 2.1 主链

- 接口：`GET https://openapi.c5game.com/merchant/account/v2/balance?app-key=...`
- 主字段：`data.moneyAmount`
- 含义：`c5余额`
- 代理：必须使用账号的 `API代理`

### 2.2 备用链

- 接口：`GET https://www.c5game.com/api/v1/account/v1/my/account`
- 备用字段：`data.balance`
- 代理：必须使用账号的 `浏览器代理`
- 请求限制：失败时不立即重试

## 3. 刷新策略

### 3.1 登录后首次补余额

登录任务成功后：

1. 最多等待 10 秒，轮询账号是否已由 open-api watch 回填 `api_key`
2. 若拿到 `api_key`，走主链拉余额
3. 若超时仍无 `api_key`，只尝试一次 session 兜底
4. 无论成功或失败，都写入下一次刷新时间

### 3.2 后续刷新

余额使用缓存展示，不阻塞账号中心列表。

每次成功或失败刷新后，都为该账号写入一个随机的下一次刷新时间，区间为 8-10 分钟。

当账号中心再次读取账号列表时：

- 若缓存未过期，直接返回缓存
- 若缓存已过期，后台异步触发一次刷新，同时先返回当前缓存

### 3.3 并发保护

同一账号同一时刻只允许一个余额刷新任务进行，避免重复刷新。

## 4. 数据模型

在账号模型中新增以下持久化字段：

- `balance_amount`
- `balance_source`
- `balance_updated_at`
- `balance_refresh_after_at`
- `balance_last_error`

用途如下：

- `balance_amount`：展示给账号中心的缓存余额
- `balance_source`：标记本次余额来源，值为 `openapi` 或 `browser_session`
- `balance_updated_at`：最近成功刷新时间
- `balance_refresh_after_at`：下次允许刷新时间
- `balance_last_error`：最近一次刷新失败原因

## 5. 后端边界

新增独立余额服务，负责：

- 判断账号是否具备 `api_key`
- 主链请求与解析
- session 兜底请求与解析
- 浏览器代理 / API代理 分流
- 刷新时间调度
- 登录成功后的首次补余额

账号中心列表聚合只负责读取缓存字段，并在必要时异步触发后台刷新。

## 6. 前端边界

前端不直接调用余额接口。

前端只做：

- 展示后端返回的余额值
- 可选展示最近来源或失败状态的简短文案

前端不做：

- 余额定时器
- `api_key` 等待逻辑
- session 兜底逻辑

## 7. 验证要点

必须验证：

1. 有 `api_key` 的账号优先显示 `moneyAmount`
2. 无 `api_key` 但已登录的账号可退到 `balance`
3. session 兜底失败时不会立即重试
4. 主链使用 `API代理`
5. 兜底链使用 `浏览器代理`
6. 登录成功后可触发首次补余额
7. 后续刷新窗口落在 8-10 分钟
