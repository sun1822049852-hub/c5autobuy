# 购买运行页库存摘要与详情设计

## 1. 目标

在现有“购买运行页”基础上，补齐用户最关心的库存运行态信息，但不改变现有购买核心逻辑。

本次只解决两个问题：

- 在购买运行页主表里直接看到每个账号当前使用的目标仓库
- 在购买运行页中查看该账号全部小仓库明细

同时保持以下边界不变：

- UI 只做展示和交互，不参与库存选择和购买判断
- 购买运行状态接口继续服务主页面轮询，不把全部仓库明细塞进主状态
- 小仓库详情通过独立接口按需获取
- 库存详情弹窗打开时读取一次快照，不做自动刷新

## 2. 已确认约束

### 2.1 主表最优先信息

购买运行页主表必须优先展示：

- 当前目标仓库
- 当前目标仓库的剩余容量 / 总容量

也就是用户打开运行页后，不需要点详情就能判断：

- 这个账号当前会往哪个仓库下单
- 这个仓库离满仓还有多少空间

### 2.2 小仓库明细查看方式

全部小仓库明细不直接铺在主页面中。

交互方式固定为：

- 用户双击购买运行页中的账号行
- 打开只读库存详情弹窗
- 弹窗内查看该账号全部小仓库快照

### 2.3 详情刷新策略

库存详情弹窗采用“一次快照”策略：

- 打开弹窗时请求一次后端详情接口
- 弹窗打开后不自动刷新
- 用户若要看更新后的库存，重新打开或手动触发后续刷新能力

本次先不增加弹窗内刷新按钮。

## 3. 范围与非目标

### 3.1 本次范围

- 购买运行状态中的账号摘要补充库存摘要字段
- 新增购买库存详情接口
- 新增库存详情弹窗
- 购买运行页账号表展示目标仓库和容量摘要
- 前后端测试覆盖库存摘要与详情读取

### 3.2 本次非目标

- 不修改库存选择算法
- 不修改购买调度器
- 不新增手工修改目标仓库功能
- 不在详情弹窗中自动刷新
- 不把全部小仓库快照写入主状态轮询接口

## 4. legacy 行为对齐点

本次展示层必须严格跟随现有购买运行时中的库存逻辑，不发明新规则。

当前后端库存行为是：

1. 账号初始化时读取本地库存快照
2. 主动拉一次远端库存快照
3. 计算各小仓库剩余容量
4. 仅保留满足阈值的可用仓库
5. 选择剩余容量最小但仍满足阈值的仓库作为目标仓库
6. 每次购买成功后，优先在本地对当前目标仓库做容量递推
7. 当前目标仓库不可用时，本地尝试切换下一个可用仓库
8. 本地无可用仓库时，再拉一次远端快照确认
9. 确认无可用仓库后，将账号移出购买池，等待恢复检查

因此本次 UI 展示必须忠实反映以下事实：

- `selected_steam_id` 就是当前目标仓库
- 当前目标仓库摘要来自当前库存快照，而不是额外推导
- 全部小仓库明细来自同一份库存快照

## 5. 后端设计

### 5.1 主状态接口继续轻量化

购买运行页主轮询仍然使用现有 `purchase runtime status` 接口。

该接口中的 `accounts[]` 新增库存摘要字段，但不直接携带全部小仓库明细。

建议新增最小字段：

- `selected_steam_id`
- `selected_inventory_remaining_capacity`
- `selected_inventory_max`

展示规则：

- 有当前目标仓库时，`selected_steam_id` 为目标仓库 ID
- 能匹配到目标仓库快照时，返回剩余容量和总容量
- 当前没有目标仓库时，上述字段都返回空值

这样主页面轮询时只取摘要，不把所有小仓库数据带到前端。

### 5.2 新增库存详情接口

新增一个按账号读取库存详情的只读接口，例如：

- `GET /purchase-runtime/accounts/{account_id}/inventory`

返回最小结构：

- `account_id`
- `display_name`
- `selected_steam_id`
- `refreshed_at`
- `last_error`
- `inventories`

其中 `inventories[]` 至少包含：

- `steamId`
- `inventory_num`
- `inventory_max`
- `remaining_capacity`

前端所需展示字段都从这组数据直接得到，不额外推导业务含义。

### 5.3 详情数据来源优先级

详情接口的数据来源优先级固定为：

1. 如果购买运行时正在运行，且该账号存在于运行时内存状态中，优先返回运行时内存快照
2. 如果当前运行时不存在该账号，则回退到库存快照仓库中的持久化数据

这样可以满足两个目标：

- 运行中的详情尽量展示最新状态
- 运行时停止后，仍能查看最后一次落盘的库存快照

### 5.4 详情接口边界

详情接口只读，不做以下动作：

- 不主动刷新远端仓库
- 不触发恢复检查
- 不修改目标仓库
- 不写回库存快照

这一步的意义只是“读当前可见状态”，不是“驱动库存逻辑”。

## 6. 前端设计

### 6.1 购买运行页主表调整

购买运行页账号表改为以“购买分配状态 + 仓库摘要”为中心。

建议列顺序：

1. 显示名
2. 购买能力
3. 购买池
4. 目标仓库
5. 容量
6. 恢复状态
7. 已购

其中：

- `目标仓库` 直接显示 `selected_steam_id`
- `容量` 显示 `remaining_capacity / inventory_max`

无目标仓库时：

- `目标仓库` 显示 `-`
- `容量` 显示 `-`

### 6.2 详情弹窗

用户双击账号表中的一行时，打开库存详情弹窗。

弹窗内容分两部分：

#### 顶部摘要

- 账号名
- 当前目标仓库
- 最近快照时间
- 最近错误

#### 下方仓库表格

建议列：

1. 仓库ID
2. 当前数量
3. 容量上限
4. 剩余容量
5. 是否当前目标
6. 是否可用

展示原则：

- 只读
- 不做编辑
- 不做自动刷新
- 打开时加载一次

### 6.3 可用仓库显示规则

“是否可用”必须与现有后端库存逻辑保持一致。

也就是：

- 当前仓库属于满足容量阈值的可用仓库集合时，显示“是”
- 否则显示“否”

这个字段应由后端在详情响应中直接给出，避免前端重复实现库存可用规则。

## 7. 错误处理

### 7.1 主表无库存摘要

如果账号当前没有目标仓库：

- 主表正常渲染
- 目标仓库和容量显示 `-`
- 不因为缺库存摘要导致整页报错

### 7.2 详情接口无数据

如果账号存在，但没有库存快照：

- 详情接口返回空明细
- 弹窗展示空状态或“暂无库存快照”

### 7.3 详情接口失败

如果详情接口请求失败：

- 只在弹窗层展示错误信息
- 不影响购买运行页主轮询
- 不影响主页面当前已加载的运行状态

### 7.4 购买能力失效或库存为空

即使账号当前：

- `purchase_capability_state = expired`
- 或 `purchase_pool_state = paused_no_inventory`

也仍允许打开详情查看最后一次库存快照与最近错误。

## 8. 测试

### 8.1 后端

至少补这些测试：

1. 主状态接口 `accounts[]` 返回目标仓库摘要字段
2. 目标仓库存在时，能正确返回 `selected_steam_id` 与 `remaining/max`
3. 当前没有目标仓库时，摘要字段返回空值
4. 库存详情接口返回当前目标仓库和全部小仓库
5. 运行时存在时，详情接口优先使用运行时内存快照
6. 运行时不存在时，详情接口回退到持久化快照

### 8.2 前端

至少补这些测试：

1. 购买运行页主表能正确显示目标仓库和容量
2. 无目标仓库时显示 `-`
3. 双击账号行会触发详情加载
4. 详情弹窗能显示全部小仓库
5. 无快照和接口失败时能显示空状态或错误状态

## 9. 实现顺序建议

1. 先补后端状态摘要字段
2. 再补库存详情 schema、service 和路由
3. 然后补前端 viewmodel 的库存摘要格式化
4. 再补购买运行页双击打开详情弹窗
5. 最后补定向测试和全量验证
