# 本地 Program Access 扩展骨架设计

日期：2026-04-13

## 1. 目标

本次只完成当前本地前后端的可扩展骨架设计与对应本地实现范围冻结，用于给未来“远端控制台 / 程序会员 / 程序登录注册”预留稳定接入位。

本次目标明确限定为：

- 当前本地程序继续可用，不影响魔尊自用
- 当前封装前，本地所有关键动作继续放行
- 本地 backend 与 frontend 内部先接上统一的 `program access` 抽象
- 未来远端控制台接入时，不需要重写关键按钮链路与本地业务主链
- 把本轮已做范围与未来待做范围写清楚，供后续对话直接接力

## 2. 当前冻结结论

### 2.1 本地数据边界

未来远端控制台只负责程序会员控制面，不负责本地业务数据。

明确不上传到远端的内容：

- C5 账号
- 查询配置
- 购买配置
- 浏览器 session / bundle
- 本地运行配置
- 本地运行时状态

因此，不再沿用“按会员把本地账号、配置、session、运行配置做多租户隔离”的方向。

### 2.2 当前阶段行为

当前阶段仍处于“封装前 / 本地自用”状态，行为冻结为：

- 可以直接进入本地界面
- 所有关键按钮继续放行
- 不引入真实程序会员拦截
- 只把将来会接会员控制面的骨架先接好

### 2.3 未来封装后的目标行为

未来正式封装并接上远端控制台后：

- 仍允许先进入本地界面
- 只在点击关键按钮时做程序会员校验
- 查询 / 购买运行内循环不反复校验程序会员
- 若当前正在运行，会员不会因一次网络抖动立刻中断
- 运行中会员续证规则采用 `24h 续证 + 72h 宽限 + 明确失效或宽限耗尽后自动暂停`

## 3. 非目标

本次明确不做：

- 远端控制台页面实现
- 程序会员管理后台实现
- 程序登录 / 注册 / 找回密码实现
- 真实 `runtime lease` 申请 / 续租实现
- 支付、续费、订单相关实现
- 本地业务数据按会员拆库 / 拆目录 / 拆 session
- 查询 / 购买核心运行逻辑重构

## 4. 总体架构

本地程序在本轮后分成三层：

1. 本地业务层
   现有账号中心、查询运行时、购买运行时、登录任务等仍按当前本地架构工作。

2. 本地 Program Access 层
   新增一层统一抽象，负责回答“当前程序访问策略是什么、关键动作是否需要经过守卫、守卫失败时返回什么标准错误”。

3. 未来远端控制面适配层
   当前只预留接口，不实现真实远端调用。后续接入远端控制台时，仅替换适配器实现，不重写业务主链。

当前默认策略是：

- `local_pass_through`
- 本地全部放行
- 仅通过 `bootstrap` 告知前端当前处于本地放行阶段

## 5. 本地 Backend 设计

### 5.1 新增 Program Access Gateway 抽象

本地 backend 需要新增一个统一抽象，建议命名为：

- `ProgramAccessGateway`

它只负责三类判断：

- 当前 program access 模式
- 某个关键动作是否需要守卫
- 若守卫失败，返回什么标准化结果

本轮默认实现为：

- `LocalPassThroughGateway`

它的规则固定为：

- `mode = local_pass_through`
- `stage = prepackaging`
- `guard_enabled = false`
- 所有关键动作直接通过

后续远端接入时，再新增：

- `RemoteLeaseGateway`

它才负责：

- 本地程序登录态读取
- 远端会员状态校验
- `runtime lease` 申请 / 续租
- 失效、宽限、暂停等决策

### 5.2 新增关键动作守卫入口

关键动作不能把未来会员判断散落到各个 use case 与页面逻辑里，必须走统一守卫层。

建议在本地 backend 的 route 层或 route 调用的统一 guard helper 中收口，当前先覆盖这些动作：

- `runtime.start`
- `runtime.switch_config`
- `account.create`
- `account.update`
- `account.delete`
- `account.login.start`
- `open_api.sync`
- `open_api.open_binding`
- `purchase_config.update`
- `query_settings.update`
- `runtime_settings.update`

本轮行为：

- 所有动作都经过统一 guard 抽象
- 但 `LocalPassThroughGateway` 永远放行

这样未来启用远端控制面时，只需替换 gateway，不重写路由分支。

### 5.3 扩展 `/app/bootstrap`

本地 backend 需要在 `GET /app/bootstrap` 中增加一段轻量 program access 摘要，例如：

```json
{
  "program_access": {
    "mode": "local_pass_through",
    "stage": "prepackaging",
    "guard_enabled": false,
    "message": "当前为本地放行模式，远端程序会员控制面尚未接入正式链路"
  }
}
```

这一段的目的不是实现会员，而是：

- 让前端知道当前运行模式
- 给未来远端模式预留稳定字段
- 避免前端通过猜测环境来决定 UI

### 5.4 Debug / Release 模式位

本地 backend 需要先引入明确的运行阶段配置位，例如：

- `local_dev`
- `packaged_release`

本轮不启用真实会员拦截，但要把切换点先放好。

冻结规则：

- 当前默认 `local_dev`
- 当前 `local_dev` 仍全部放行
- 未来远端会员接入后，`debug/dev` 允许开发管理员绕过口子
- 未来 `release` 打包时必须彻底封口，不能只隐藏 UI

## 6. 本地 Frontend 设计

### 6.1 读取并展示 Program Access 状态

前端从 `/app/bootstrap` 读取 `program_access`，增加一个轻提示区或状态条。

本轮只需要明确告诉用户：

- 当前是本地放行模式
- 当前会员控制面尚未正式接入

该提示不阻断任何现有操作。

### 6.2 关键动作反馈统一出口

虽然本轮 backend 不拦截，但前端应统一准备好关键动作失败提示出口，避免未来接入会员控制面时把错误处理散在各页。

后续若 backend 返回标准错误码，前端统一映射提示：

- `program_auth_required`
- `program_membership_expired`
- `program_membership_service_unavailable`
- `program_feature_not_enabled`
- `program_guard_bypassed_dev_only`

本轮基本不会触发这些错误，但前端结构上要预留统一承接点。

### 6.3 预留程序登录入口占位

本轮不实现真实程序登录 / 注册，但前端可以预留一个极轻量入口位或状态位，用于后续接入：

- 程序会员登录入口按钮占位
- 程序会员状态文案占位

本轮不接真实交互，不强行导流。

## 7. 关键动作矩阵

未来正式接入远端控制面后，关键动作矩阵冻结如下：

- `开始扫货`：受控
- `切换 / 应用配置`：受控
- `新增 / 编辑 / 删除账号`：受控
- `发起账号登录`：受控
- `打开 / 同步 API 白名单`：受控
- `修改购买仓库`：受控
- `修改购买设置`：受控
- `修改查询设置`：受控

始终不应被程序会员拦截的动作：

- 查看页面
- 查看日志
- 查看状态
- 手动停止当前运行

本轮虽然继续全部放行，但动作边界应按此矩阵冻结。

## 8. 统一错误码冻结

未来远端控制面接入后，本地 backend 统一返回以下错误码：

- `program_auth_required`
- `program_membership_expired`
- `program_membership_service_unavailable`
- `program_feature_not_enabled`
- `program_guard_bypassed_dev_only`

本轮冻结其名称和语义，但不要求真实触发。

## 9. 未来远端控制面预留接口

本轮不实现远端控制台，但必须把未来会接入的边界写清楚。

远端控制面未来只负责：

- 程序会员用户
- 程序会员状态
- 程序登录 / 注册 / 找回密码
- `runtime lease` 签发与续租
- 审计与吊销

未来客户端到远端的接口类别仅预留，不落实现：

- 程序登录
- 刷新程序会话
- 查询当前会员摘要
- 申请 / 续租 `runtime lease`

本轮的本地 `ProgramAccessGateway` 就是为这层对接预留的唯一入口。

## 10. 未来 Runtime Lease 规则冻结

虽然本轮不实现真实 `runtime lease`，但为了防止后续对话跑偏，其规则先冻结如下：

- 关键按钮点击时校验程序会员
- 运行中不在查询 / 购买内循环里反复校验
- 自最近一次成功续证起，`24h` 后进入续证窗口
- 若远端明确返回会员失效，则当前运行自动暂停
- 若只是网络抖动或远端暂时不可达，则进入 `72h` 宽限
- 宽限期内不允许新的关键动作
- 宽限耗尽仍未续证成功，则自动暂停

这些规则只作为未来控制面阶段的冻结约束，不属于本轮实现范围。

## 11. 本轮实现边界

### 11.1 本轮要实现

- 本地 backend 的 `ProgramAccessGateway` 抽象
- `LocalPassThroughGateway` 默认实现
- 关键动作统一 guard 入口
- `/app/bootstrap` 暴露 `program_access`
- frontend 读取并展示 `program_access`
- frontend 统一预留关键动作错误反馈承接点
- frontend 预留程序登录状态 / 入口占位

### 11.2 本轮不实现

- 真实远端控制台
- 真实程序会员登录 / 注册
- 真实会员状态校验
- 真实 `runtime lease`
- 真实自动暂停与宽限逻辑

## 12. 测试范围

本轮测试只验证“骨架已接好、行为不变”：

- `/app/bootstrap` 包含 `program_access`
- 当前本地放行模式下，关键动作仍可成功执行
- 关键动作已统一接到 guard 层
- frontend 能显示本地放行模式提示
- 当前本地模式不影响现有账号中心、查询运行时、购买运行时

不测试：

- 真实远端登录注册
- 真实会员过期
- 真实 lease 续租
- 真实 release 封口

## 13. 对后续对话的交接要求

后续若开启新的对话推进远端控制台 / 会员 / 程序登录注册，必须以本 spec 为边界继续：

- 不回到“本地业务数据按会员多租户隔离”的旧方向
- 不把程序会员校验塞进查询 / 购买内循环
- 优先复用本轮新增的 `ProgramAccessGateway` 与 guard 入口
- 远端控制台只做程序会员控制面，不扩成业务主后台

## 14. 决策摘要

本次冻结以下决策：

- 当前封装前，本地继续全部放行
- 本轮只实现本地前后端的 program access 扩展骨架
- 远端控制台 / 会员 / 程序登录注册只留边界，不在本轮实现
- 本地业务数据继续完全留在用户机器，不做会员多租户隔离
- 未来正式封装后，关键按钮才进入远端程序会员控制链
- 未来运行中控制采用 `runtime lease`，并遵守 `24h 续证 + 72h 宽限 + 自动暂停` 规则

这份文档的作用不是宣布“会员系统已经开始实现”，而是明确：

- 当前这轮要把本地程序改造成“可接会员控制面”
- 后续那一轮再去做真实远端控制台、会员管理与程序登录注册
