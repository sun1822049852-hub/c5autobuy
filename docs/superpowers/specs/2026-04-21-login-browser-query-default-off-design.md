# 登录后浏览器扫货默认关闭设计

## 背景

账号中心当前流程是先创建账号壳，再通过扫码登录把 `c5_user_id`、`cookie_raw` 等登录结果回写到账号。现状中，浏览器扫货能力会在登录成功后保持开启状态，导致新登录成功的账号会立刻处于“浏览器扫货已启用”状态。

用户确认的新约束不是“所有登录成功都强制关闭”，而是更窄的产品行为：

- 仅当某个账号第一次成功绑定登录身份时，浏览器扫货默认关闭。
- 已有登录身份的老账号再次登录时，不得覆盖它当前的浏览器扫货开关。
- 登录结果如果并回已有老账号，也不得覆盖该老账号原有的浏览器扫货开关。

## 目标

- 把“首次绑定成功后默认关闭浏览器扫货”沉到后端真实回写链路，而不是只改前端显示。
- 保持登录成功链路、重复账号归并链路、白名单复用链路、购买能力绑定链路的既有恢复结果不变。
- 不新增 schema、不新增前后端协议字段、不引入新的状态码。

## 非目标

- 不把“新建空账号”本身的默认浏览器扫货状态改为关闭。
- 不改变 API 查询开关逻辑。
- 不改白名单页 `open-api` 复用优先级。
- 不改购买配置、购买池、库存刷新或登录成功后的购买绑定逻辑。
- 不新增“首次登录默认关闭”专属展示文案。

## 方案比较

### 方案 A：只在首次绑定成功时关闭浏览器扫货（采用）

在登录成功回写前，判断目标账号此前是否已经绑定过登录身份。若此前未绑定，则在成功回写登录结果时同步写入：

- `token_enabled=False`
- `browser_query_disabled_reason="manual_disabled"`

若此前已经绑定，则保留现有 `token_enabled` 与 `browser_query_disabled_reason`，不做覆盖。

优点：

- 精准匹配“方案二”的产品边界。
- 只改首次绑定分支，不误伤老账号重登。
- 无需改前端协议和数据结构。

代价：

- 继续复用 `manual_disabled` 作为关闭原因，界面上会显示“手动禁用”，语义不是最精确。

### 方案 B：直接把创建账号默认值改为关闭

在账号壳创建时直接把浏览器扫货默认置为关闭。

不采用原因：

- 这会改变“未登录空账号”的默认状态，扩大影响范围。
- 用户要求的是“登录后默认关闭”，不是“新建账号默认关闭”。

### 方案 C：只改前端展示为关闭

仅在前端归一化层把首次登录后的账号显示为关闭，但后端真实 `token_enabled` 不变。

不采用原因：

- 前端显示与后端真实运行状态会分叉，风险最高。

## 设计细节

### 1. 普通登录成功回写

登录任务在确定最终写回目标账号后，基于“写回前的账号状态”判断这是否是首次绑定：

- 若目标账号此前没有有效登录身份（例如 `c5_user_id` 为空），则视为首次绑定。
- 首次绑定时，除现有登录成功字段外，额外写入：
  - `token_enabled=False`
  - `browser_query_disabled_reason="manual_disabled"`
- 若目标账号此前已经绑定过登录身份，则不写这两个字段。

这一定义覆盖两种情况：

- 普通空账号首次登录成功并写回自身。
- API-only 来源登录后命中空的新登录账号并写回新账号。

### 2. 命中已有老账号时的归并

当登录结果命中仓库里已存在的同 `c5_user_id` 账号时，最终写回目标是该老账号。此时：

- 继续沿用现有归并规则。
- 不覆盖该老账号已有的浏览器扫货开关。

原因是这类账号不是“首次绑定”，只是再次登录或重新接管会话。

### 3. 冲突分支

当用户在登录冲突里选择：

- `create_new_account`
- `replace_with_new_account`

系统都会生成一个全新的登录账号并写入捕获到的登录结果。新账号天然属于首次绑定，因此默认写入：

- `token_enabled=False`
- `browser_query_disabled_reason="manual_disabled"`

旧账号是否保留、是否被替换，继续维持现有冲突处理语义，不额外改动。

## 受影响模块

- 登录成功回写链路
- 登录冲突落地新账号链路
- 后端登录相关测试

## 明确不修改的模块

- 白名单页打开链路与 session/profile 复用优先级
- 登录成功后 profile 持久化时机
- 购买能力绑定与库存刷新
- 账号查询模式的接口契约与前端切换交互

## 状态码与展示

本次继续复用现有关闭原因：

- `browser_query_disabled_reason="manual_disabled"`

原因：

- 当前前后端已有完整的展示和切换逻辑。
- 不增加 schema、映射和文案扩散。

已知折中：

- 界面会显示“手动禁用”，而实际来源是“首次登录默认关闭”。

后续若产品需要区分来源，可再单独引入专用 reason code，但不属于本轮范围。

## 测试策略

按 TDD 执行，先补失败用例，再补实现。至少覆盖以下场景：

1. 空账号首次登录成功后，浏览器扫货默认关闭。
2. 已绑定老账号再次登录成功后，浏览器扫货开关保持原值。
3. API-only 来源登录并新建登录账号时，新账号默认关闭浏览器扫货。
4. 登录结果并回已有老账号时，老账号原有浏览器扫货开关保持不变。
5. 冲突选择 `create_new_account` 时，新账号默认关闭浏览器扫货。
6. 冲突选择 `replace_with_new_account` 时，新账号默认关闭浏览器扫货。

验证层级采用中风险策略：

- 先跑新增/修改的登录链路测试，确认红绿过程成立。
- 再跑登录冲突相关测试。
- 回读改动区，确认没有扩大到白名单复用、重复账号归并和老账号重登行为。

## 回退策略

若行为不符合预期，只需撤回“首次绑定时额外写入浏览器扫货关闭状态”的回写逻辑：

- 不涉及 schema 回退。
- 不涉及前端协议回退。
- 不涉及登录/白名单/购买链的其它关键恢复逻辑。

## 用户确认结论

用户已明确确认以下设计结论：

- 采用“方案二”的产品边界：不是所有登录成功都关闭，只在首次绑定成功时默认关闭。
- 采用方案 A：只改首次绑定成功的后端回写点。
- 冲突新建账号与替换账号都视为首次绑定，默认关闭浏览器扫货。
