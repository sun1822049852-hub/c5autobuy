# 桌面 Web 运行页紧凑化重排设计

日期：2026-03-20

## 1. 目标

为新的 `app_desktop_web` 做一轮聚焦“信息密度”和“运行态可读性”的页面重排，解决当前三大页面顶部占高、购买页主次倒置的问题。

本次设计目标：

- 移除账号中心、查询系统、购买系统的 Hero 大标题区
- 保留现有功能入口与运行逻辑，不因 UI 重排改变行为
- 让购买系统第一眼聚焦“当前配置 + 商品运行表现”
- 保留右下角悬浮 `开始扫货 / 停止扫货`
- 把购买页商品区从 accordion 改为平铺监控行
- 降低辅助区权重，让账号监控和最近事件退到次级

本次设计是展示层重排，不是运行链路重写。

## 2. 当前问题

当前桌面 Web 虽然已经拆出三大页面，但存在以下问题：

1. 三个页面都保留了占高较大的 Hero 头部，浪费纵向空间。
2. 购买页顶部同时存在 Hero 和大块运行摘要，进入页面后主信息下沉过深。
3. 购买页商品列表仍是可展开卡片，运行时无法一眼看全关键指标。
4. 购买页顶部仍显示：
   - `队列中`
   - `运行代号 / 购买代号`
   这些信息对当前人工盯盘优先级不高。
5. 购买页当前视觉主次不对：
   - 商品运行表现应该是主位
   - 账号监控和最近事件应该是辅助位

## 3. 已确认的用户决策

以下内容已经确认：

- 三个页面都移除大标题 Hero 区
- 购买页移除 `队列中` 展示
- 购买页移除 `购买代号 / 运行代号` 展示
- 购买页顶部改为一条紧凑运行栏
- 运行栏左侧显示当前配置
- `选择配置 / 切换配置` 放在当前配置右侧
- `累计购买` 放在同一行，位于配置动作之后
- 右下角悬浮 `开始扫货 / 停止扫货` 保留，不并入顶部
- 购买页商品区改为平铺监控行，不再用 accordion
- 每个商品左侧显示：
  - 名称
  - 磨损
  - 价格信息
- 每个商品右侧显示：
  - `已购`
  - `命中`
  - `回执`
  - `查询次数`
- 购买页现有账号监控和最近事件保留，但降为辅助区
- 本轮重排只改展示结构，不借机修改运行逻辑

## 4. 方案比较

### 方案 A：保留现有页面骨架，仅压缩头部并重排行情区

优点：

- 风险最低
- 不改变现有操作路径
- 最符合“先可用，再精修”的节奏
- 可以最大限度避免 UI 改动带出运行时回归

缺点：

- 账号监控与最近事件仍会和主区共处一个页面
- 视觉冲击力不强，但足够稳

### 方案 B：购买页改为上下双工作区

优点：

- 主区和辅助区边界更明显
- 更像专门的运行台

缺点：

- 布局改动更大
- 需要更强的响应式重排
- 容易引入额外样式回归

### 方案 C：购买页改为 dashboard 卡片墙

优点：

- 视觉最强
- 指标表达更显眼

缺点：

- 容易做成“好看但不顺手”
- 不适合当前高频人工扫视的使用场景
- 与现有样式体系差异过大

### 结论

采用方案 A。

原因：

- 它在不改变运行逻辑的前提下，能最快把页面重心拉回正确位置。

## 5. 设计范围

本次设计覆盖以下页面：

1. `AccountCenterPage`
2. `QuerySystemPage`
3. `PurchaseSystemPage`

本次设计不覆盖：

- 购买运行逻辑
- 查询运行逻辑
- 配置保存逻辑
- 账号登录逻辑
- 后端接口字段语义

## 6. 页面级设计

### 6.1 账号中心

#### 现状

账号中心当前最上方有一整块 Hero，承载：

- 页面标题
- 页面描述
- 后端状态
- 刷新按钮

#### 重排方案

移除整块 Hero。

页面进入后直接显示：

1. `OverviewCards`
2. 列表工具栏
3. 账号表格
4. 状态条

其中：

- `后端状态` 不再单独占据顶部横幅
- `刷新` 仍保留为可见操作
- 两者并入现有工具区即可

#### 目标效果

- 减少纵向占高
- 更快进入账号列表操作
- 保持账号中心当前交互路径不变

### 6.2 查询系统

#### 现状

查询系统当前最上方也有 Hero，承载：

- 页面标题
- 页面描述
- 后端状态

#### 重排方案

移除整块 Hero。

页面主结构保持不变：

- 左侧 `QueryConfigNav`
- 右侧 `QueryWorkbenchHeader + 商品区 + 保存栏`

后端状态不再占一整块横幅，而是并入：

- `QueryWorkbenchHeader`
  或
- `QueryConfigNav` 顶部轻量区

后续实现时优先选择对现有组件侵入最小的一种。

#### 目标效果

- 保持“左侧配置导航，右侧工作台”的现有心智
- 去掉欢迎页式头部，直接进入配置管理

### 6.3 购买系统

#### 现状

购买系统当前同时存在：

- 页面 Hero
- 大块 `PurchaseRuntimeHeader`
- accordion 商品卡片
- 右下角悬浮动作条

主要问题是进入页面后主信息过深，商品区又不能一眼读完。

#### 重排方案

购买页自上而下调整为四层：

1. 紧凑运行栏
2. 当前配置商品清单
3. 辅助区
4. 右下角悬浮动作条

##### 第一层：紧凑运行栏

运行栏保留单行主信息，不再用大摘要卡。

建议包含：

- 当前配置名
- 运行状态文案
- `选择配置 / 切换配置`
- `累计购买`

运行栏中明确移除：

- `队列中`
- `运行代号 / 购买代号`

##### 第二层：当前配置商品清单

商品区直接显示当前配置下的商品监控行。

每个商品一行，左右分区：

- 左侧：商品是什么
- 右侧：当前跑成什么样

左侧固定显示：

- `item_name`
- `detail_min_wear ~ detail_max_wear`
- `价格 <= max_price`

右侧固定显示四个统计位：

- `已购 = purchase_success_count`
- `命中 = matched_product_count`
- `回执 = purchase_failed_count`
- `查询次数 = query_execution_count`

商品区不再采用 accordion，不再要求用户点击展开才能看到关键指标。

##### 第三层：辅助区

辅助区保留现有：

- `PurchaseAccountTable`
- `PurchaseRecentEvents`

但它们降级为次级信息：

- 位置在商品区之后或右侧次栏
- 标题样式降低权重
- 不再先于商品区抢占注意力

##### 第四层：悬浮动作条

右下角悬浮动作条继续保留：

- 当前配置显示
- `选择配置 / 切换配置`
- `开始扫货 / 停止扫货`

这里不改交互位置，只做与新顶部信息的重复收敛和样式协调。

## 7. 购买页商品行设计

### 7.1 结构

商品行采用紧凑双列布局：

- 左侧信息列
- 右侧统计列

不再显示：

- 展开箭头
- `展开详情 / 收起详情`
- 重复的详情卡片

### 7.2 左侧信息列

从上到下显示：

1. 商品名
2. 磨损信息
3. 价格信息

推荐文案形式：

- `磨损 0.01 ~ 0.07`
- `价格 <= 128.00`

### 7.3 右侧统计列

固定顺序：

1. `已购`
2. `命中`
3. `回执`
4. `查询次数`

每个统计位采用小号数字块，保证横向可扫读。

### 7.4 信息优先级

优先级冻结为：

1. 商品名
2. 磨损 / 价格约束
3. 已购 / 命中 / 回执 / 查询次数

不在商品行中展示：

- 购买代号
- 队列指标
- 展开态提示

## 8. 状态与空态设计

### 8.1 未选择配置

购买页顶部显示：

- `未选择配置`

商品区显示空态提示：

- 当前没有可展示的商品统计

右下角 `开始扫货` 保持禁用。

### 8.2 已选择但未运行

显示：

- 当前配置名
- 当前配置商品静态信息
- `开始扫货` 可点击

运行状态文案保持未运行语义。

### 8.3 运行中

显示：

- 当前配置名
- 当前运行状态文案
- 商品行动态刷新统计
- 右下角按钮切换为 `停止扫货`

### 8.4 加载失败

继续复用当前错误展示区，不新增新的错误通道。

## 9. 数据契约与组件边界

本次设计不要求后端新增字段。

购买页现有字段已足够支持重排：

- `configDisplayName`
- `runtimeMessage`
- `totalPurchasedCount`
- `status.item_rows`
- `status.accounts`
- `status.recent_events`

本次只是改变这些字段的展示位置和展示密度。

建议组件边界调整为：

- `PurchaseSystemPage`
  - 负责新的页面骨架
- `PurchaseRuntimeHeader`
  - 改造成紧凑运行栏，或由更轻量的新组件替代
- `PurchaseItemPanel`
  - 改造成非 accordion 的商品监控行
- `PurchaseRuntimeActions`
  - 保留悬浮操作，不改变核心交互

## 10. 响应式设计

### 10.1 桌面宽屏

购买页商品行保持：

- 左侧商品信息
- 右侧四个统计位横排

辅助区可维持：

- 商品主列
- 账号监控次列

最近事件继续位于主列下方或次列中，按现有样式体系择优。

### 10.2 中窄屏

购买页商品行改为两层：

1. 上层：商品名 + 磨损 + 价格
2. 下层：四个统计位

统计位顺序不变，防止人工扫读位置漂移。

### 10.3 小屏

右下角悬浮动作条继续全宽贴底显示，维持现有移动端策略。

## 11. 非目标

本次设计明确不做以下内容：

- 不新增购买页的查询队列监控替代物
- 不新增购买页过滤、排序、搜索
- 不把悬浮动作条并入顶部栏
- 不重做账号监控表格结构
- 不改后端状态轮询频率
- 不改购买系统运行语义

## 12. 风险与约束

### 12.1 最大风险

UI 重排时错误牵动运行逻辑。

#### 约束

- 只允许修改页面布局、组件展示和样式
- 不允许借本轮重排调整启动、停止、切换配置逻辑

### 12.2 次级风险

购买页顶部与悬浮条信息重复过多。

#### 约束

- 顶部强调状态和累计
- 悬浮条强调动作和当前配置
- 不在两处重复展示低优先级指标

### 12.3 兼容性风险

accordion 改平铺后，移动端容易拥挤。

#### 约束

- 商品统计位必须支持自动换行
- 小屏时改上下两层，不允许横向溢出

## 13. 测试重点

后续实现至少需要覆盖：

1. 三个页面都不再渲染 Hero 大标题区
2. 账号中心移除 Hero 后，刷新和后端状态仍可见且可用
3. 查询系统移除 Hero 后，配置管理和保存流程不回归
4. 购买页不再展示 `队列中`
5. 购买页不再展示 `运行代号 / 购买代号`
6. 购买页顶部能显示当前配置、状态文案和累计购买
7. 购买页商品区默认直接显示：
   - 名称
   - 磨损
   - 价格
   - 已购
   - 命中
   - 回执
   - 查询次数
8. 商品区不再依赖 accordion 才能看到关键指标
9. 右下角悬浮 `开始扫货 / 停止扫货` 仍可正常工作
10. 未选择配置、未运行、运行中三种状态文案与按钮状态不回归
11. 中窄屏下商品行不会横向溢出

## 14. 结论

这次重排的核心不是“做一个新页面”，而是把现有页面的信息权重拨回正确位置：

- 账号中心直接进入管理
- 查询系统直接进入配置工作台
- 购买系统直接进入“当前配置 + 商品表现”

其中购买页是本轮重点：

- 去掉大头部
- 去掉低优先级运行指标
- 保留悬浮主动作
- 把商品监控行变成默认可读

只有先把这些展示边界锁死，后续实现与手测精修才不会再一边改 UI，一边把运行语义带偏。
