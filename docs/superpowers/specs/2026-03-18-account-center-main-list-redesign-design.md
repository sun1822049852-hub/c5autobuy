# 账号中心主列表重构设计

日期：2026-03-18

## 1. 目标

把当前账号中心从“左侧只负责展示，右侧详情区负责操作”的主从结构，重构为“列表即主操作区”的账号管理界面。

本次改动的目标是：

- 左侧账号列表成为主要操作入口。
- 列表固定展示 `C5昵称 / API Key / 购买状态 / 代理` 四列。
- 用户直接点击对应单元格即可完成备注、API Key、代理和购买配置相关操作。
- 删除账号通过右键菜单触发，并保留二次确认。
- 保持现有 Python 后端登录、查询、购买逻辑不变，只重构账号中心这一层交互与必要的数据聚合。

## 2. 当前问题

当前账号中心存在以下交互问题：

1. 左侧表格只起到选中作用，真正操作都堆在右侧详情区，路径过长。
2. 当前列表列定义偏技术视角，和用户真正关心的“是否可买、当前仓库、是否有 Key、代理是什么”不一致。
3. 备注、API Key、代理、登录、购买配置分散在不同按钮和详情块里，用户必须先选中，再看右侧，再找按钮。
4. “添加账号”与“删除账号”都不是贴着列表使用，管理批量账号时效率低。
5. 当前后端账号列表接口没有直接提供“当前仓库”这类列表级展示字段，导致前端无法稳定表达“可购买且已伴随仓库”的业务语义。

## 3. 已确认的用户决策

以下需求已经在讨论中确认，本设计以这些决策为准：

- 账号中心主列表固定为四列：
  - `C5昵称`
  - `API Key`
  - `购买状态`
  - `代理`
- 不再单独展示“登录状态”列，登录语义并入“购买状态”。
- `购买状态` 的显示优先级固定为：
  - `未登录`
  - `禁用`
  - `库存已满`
  - `当前仓库内容`
- 账号登录后会自动选择一个仓库，并且该仓库会伴随账号持续存在。
- `C5昵称` 点击后编辑备注，不修改平台真实昵称。
- `API Key` 点击后只修改 Key，不强制重新登录。
- `代理` 点击后编辑完整代理；代理若被修改，保存后需要重新登录。
- `添加账号` 按钮位于列表表头区域右上角，不单独保留右侧栏。
- 右键某一行账号时，菜单中提供 `删除账号`，点击后仍需二次确认。

## 4. 范围与非目标

### 4.1 本次范围

- 重构账号中心主界面结构。
- 重写账号列表列定义、显示规则和单元格点击行为。
- 补充列表表头右上角 `添加账号` 按钮。
- 增加账号行右键菜单和删除确认。
- 新增或复用轻量配置弹框，覆盖：
  - 备注编辑
  - API Key 编辑
  - 代理编辑
  - 购买配置
- 为列表展示补齐“当前仓库”和“购买状态”所需的数据聚合。
- 为购买配置补齐“禁用购买能力”和“切换当前仓库”的后端支持。

### 4.2 非目标

- 不把整个桌面程序迁移成 Web 前端。
- 不修改底层 Python 登录、查询、购买执行逻辑。
- 不改 Selenium 登录实现。
- 不把账号中心做成完整的浏览器页面；本次仍以当前桌面端代码结构为基础实现。
- 不新增“仅本次登录临时代理但不落库”的能力。

说明：

用户已经明确询问过“前端是否能做成浏览器式交互，而后端继续保持 Python”。答案是可以，但那属于后续演进方向，不是本次账号中心重构的实现范围。本次设计应尽量让数据接口和前端状态建模保持清晰，以便未来需要时平滑迁移到本地 Web UI。

## 5. 方案比较

### 方案 A：保留主从布局，只优化右侧详情区

- 左侧列表继续以“选中”为主。
- 右侧详情区继续承担登录、编辑、清除购买能力等主要操作。
- 仅调整列表列定义和少量按钮位置。

优点：

- 改动面最小。
- 复用现有详情面板成本低。

缺点：

- 仍然保留“先选中，再去右侧操作”的多步路径。
- 与用户想要的“列表就是主操作区”不一致。
- 后续即使改成 Web UI，也仍然是偏重的主从面板模型。

### 方案 B：交互式主列表 + 轻量弹框 + 行右键菜单

- 列表直接承担主要操作。
- 单元格点击即进入对应配置。
- 表头右上角放 `添加账号`。
- 删除走右键菜单。
- 不再保留常驻右侧详情栏。

优点：

- 与用户确认的目标完全一致。
- 主操作集中，路径最短。
- 更容易做成桌面端和未来本地 Web UI 都能复用的交互模型。
- 前端职责清晰：列表负责展示与分发动作，弹框负责具体修改。

缺点：

- 需要补齐单元格点击、右键菜单、列表数据聚合和购买配置写入能力。
- 现有详情面板相关代码需要降级或移除。

### 方案 C：卡片式账号管理页

- 不再用表格，改成账号卡片列表。
- 每张卡片上展示昵称、Key、仓库、代理和操作按钮。

优点：

- 视觉上更直观，也更接近浏览器式信息卡片。

缺点：

- 不适合高密度账号管理。
- 与当前代码结构差异过大。
- 对桌面端批量查看和对比不如表格高效。

### 结论

采用方案 B。

它最符合已确认需求，也能在不触碰底层业务逻辑的前提下，把账号中心真正改成高频操作场景可用的主工作区。

## 6. 交互设计

### 6.1 整体布局

账号中心主体收敛为一个列表区：

- 顶部保留状态提示区域。
- 列表表头区域右上角提供 `添加账号` 按钮。
- 不再保留右侧常驻详情栏。
- 列表支持单行选中，用于高亮当前账号和支撑右键菜单。

### 6.2 列定义

列表固定四列：

1. `C5昵称`
2. `API Key`
3. `购买状态`
4. `代理`

每列都应表现为可点击内容，而不是纯只读文本。

### 6.3 `C5昵称` 列

显示规则：

- 优先显示 `remark_name`
- 若为空，则回退到 `c5_nick_name`
- 再为空则回退到 `default_name`

点击行为：

- 打开轻量备注编辑弹框。
- 仅修改 `remark_name`。
- 保存成功后刷新当前行显示。

### 6.4 `API Key` 列

显示规则：

- 有 Key 显示 `有`
- 无 Key 显示 `无`

点击行为：

- 打开轻量 API Key 编辑弹框。
- 允许新增、修改、清空 Key。
- 保存后直接更新账号。
- 不强制重新登录。

如果后续仍需保留查询模式开关，建议在同一弹框内复用现有 `_AccountDialogBase` 的查询开关逻辑，而不是再复制一套 API Key 相关状态管理。

### 6.5 `代理` 列

显示规则：

- 有代理时显示完整代理字符串
- 无代理时显示 `直连`

点击行为：

- 打开轻量代理编辑弹框。
- 保存时比较新旧代理：
  - 若代理未变化：仅关闭弹框，不触发登录
  - 若代理有变化：先保存账号代理，再触发重新登录

这里沿用已经落地的“登录前代理确认/提交流程”设计，不重新发明另一套代理更新逻辑。

### 6.6 `购买状态` 列

显示优先级固定如下：

1. `未登录`
2. `禁用`
3. `库存已满`
4. `当前仓库内容`

具体判定规则：

- 只要账号未登录或登录失效，就显示 `未登录`
  - 判定来源：`purchase_capability_state != bound`
  - 或 `purchase_pool_state == paused_auth_invalid`
- 否则，如果账号被禁用购买，显示 `禁用`
  - 判定来源：`disabled == true`
- 否则，如果当前没有可用仓库，显示 `库存已满`
  - 判定来源：`purchase_pool_state == paused_no_inventory`
  - 或没有选中的有效仓库
- 否则，直接显示当前仓库内容
  - 当前阶段可先显示 `selected_steam_id`
  - 若后端后续能提供更友好的仓库展示名，只替换显示文本，不改交互规则

点击行为：

- 点击 `未登录`：直接拉起登录流程
- 点击 `禁用`：打开购买配置弹框，可重新启用
- 点击 `库存已满`：打开购买配置弹框，但满仓库不可选
- 点击当前仓库内容：打开购买配置弹框，可切换仓库

### 6.7 购买配置弹框

购买配置弹框负责两件事：

- 选择当前仓库
- 开启/关闭账号购买能力

弹框内容建议包括：

- 当前账号名称
- 当前购买状态
- `参与购买` 开关
- 仓库列表
  - 当前仓库高亮
  - 满仓或不可用仓库置灰，不允许选择
  - 若账号未登录，不展示仓库选择，直接引导登录

保存规则：

- 当用户关闭购买能力时：
  - 设置 `disabled = true`
  - 该账号不再参与购买任务分配
- 当用户开启购买能力时：
  - 设置 `disabled = false`
  - 若当前已有有效仓库，则可重新参与购买
- 当用户切换仓库时：
  - 保存新的 `selected_steam_id`
  - 若运行时正在运行，应同时更新运行时中的当前仓库

### 6.8 表头按钮

列表表头区域右上角固定提供 `添加账号` 按钮。

点击后：

- 打开现有 `CreateAccountDialog` 或其轻量复用版本
- 保存成功后刷新列表

### 6.9 右键菜单

右键某一行账号时，弹出上下文菜单，至少包含：

- `删除账号`

点击 `删除账号` 后：

- 弹出确认框
- 用户确认后才真正删除
- 删除成功后从列表移除该行
- 删除失败则提示错误并保持原列表

## 7. 数据与接口设计

## 7.1 列表读模型

当前 `GET /accounts` 只返回账号实体字段，不包含当前仓库、列表级购买状态文案等主列表所需信息。

为了避免前端自己拼接多份来源，也避免为每个账号单独请求库存详情，建议新增一个账号中心专用读模型接口，例如：

- `GET /account-center/accounts`

返回内容应至少包括：

- `account_id`
- `remark_name`
- `c5_nick_name`
- `default_name`
- `display_name`
- `api_key_present`
- `api_key`
- `proxy_mode`
- `proxy_url`
- `proxy_display`
- `purchase_capability_state`
- `purchase_pool_state`
- `disabled`
- `selected_steam_id`
- `selected_warehouse_text`
- `purchase_status_code`
- `purchase_status_text`

其中：

- `selected_steam_id` / `selected_warehouse_text` 优先从运行时状态获取
- 若运行时没有该账号，则回退到库存快照
- `purchase_status_code` / `purchase_status_text` 在后端统一计算，前端只负责展示和按 code 分发动作

这样可以把“账号中心主列表需要什么”明确定义成一个稳定读模型，未来若切换到本地 Web UI，也可以直接复用这份接口。

### 7.2 购买配置读取

仓库选择弹框可以继续复用现有库存详情读取能力：

- `GET /purchase-runtime/accounts/{account_id}/inventory`

该接口已经能返回：

- `selected_steam_id`
- 仓库列表
- 每个仓库的 `is_selected`
- 每个仓库的 `is_available`

这足以支撑购买配置弹框的仓库展示。

如果需要在未运行时也稳定读取仓库信息，则继续沿用其“运行时优先、快照回退”的现有逻辑。

### 7.3 购买配置写入

当前后端没有正式的“切换当前仓库”和“修改 disabled”的专用命令，因此需要新增写接口，例如：

- `PATCH /accounts/{account_id}/purchase-config`

payload 建议包含：

- `disabled: bool`
- `selected_steam_id: str | null`

写入规则：

- 更新账号的 `disabled`
- 更新库存快照中的 `selected_steam_id`
- 若购买运行时正在运行且账号存在于运行时状态中，同时同步运行时中的当前仓库

校验规则：

- 若账号未登录，则不允许写入仓库选择
- 若仓库不存在，返回校验错误
- 若仓库不可用或已满，不允许选中
- 若账号被禁用，可以保留当前仓库，但不参与调度

### 7.4 现有账号更新接口复用

以下动作继续复用现有 `PATCH /accounts/{account_id}`：

- 修改备注
- 修改 API Key
- 修改代理

这样可以避免为基础字段再拆无意义的新接口。

## 8. 前端组件拆分

### 8.1 列表组件

当前 `AccountTableWidget` 只负责简单表格展示。

重构后建议升级为支持以下能力：

- 四列固定表头
- 单元格点击事件分发
- 行右键菜单
- 表头动作按钮区域
- 当前行选中高亮

如果 `QTableWidget` 的表头扩展成本过高，可以在表格外层增加一个轻量头部容器，把标题与 `添加账号` 按钮放在同一行，而不是为此再引入更重的自定义表格框架。

### 8.2 轻量编辑弹框

建议新增或拆分以下轻量弹框：

- `RemarkEditDialog`
- `ApiKeyDialog`
- `ProxyEditDialog`
- `PurchaseConfigDialog`

这些弹框不应各自复制代理、Key、查询开关等字段逻辑。

应优先复用现有 `_AccountDialogBase` 中已经稳定的输入部件和 payload 构造逻辑，只把无关字段隐藏掉，避免新旧两套账号编辑规则出现分叉。

### 8.3 窗口与控制器职责

`AccountCenterWindow` 负责：

- 渲染主列表
- 分发表头按钮、单元格点击和右键菜单事件
- 打开对应弹框

`AccountCenterController` 负责：

- 加载账号中心读模型
- 编排备注/API Key/代理/购买配置的更新动作
- 处理“代理修改后重登”
- 处理删除账号
- 刷新列表与状态提示

`AccountCenterViewModel` 负责：

- 保存账号中心行数据
- 提供当前选中账号
- 不再把复杂购买状态文案散落在多个 formatter 中重复计算

## 9. 错误处理

- 备注更新失败：提示失败，不刷新该行
- API Key 更新失败：提示失败，不改列表显示
- 代理更新失败：不触发重新登录
- 代理更新成功但重新登录失败：保留新代理，购买状态回到 `未登录`
- 购买配置保存失败：不修改列表状态
- 删除失败：提示失败，不移除该行
- 登录拉起失败：沿用现有错误发布方式

## 11. 风险

主要风险如下：

1. 如果继续直接复用旧详情面板思路，容易把新列表又做回“只是入口，真正逻辑还在别处”。
2. 如果购买状态仍由多个前端 formatter 和多个窗口各自计算，后续一定会出现状态不一致。
3. 当前后端没有正式的“切仓”入口，如果不补齐写接口，前端的购买配置只能做成假交互。
4. 如果代理编辑和重登逻辑再次复制一套，容易和已落地的登录前代理流程产生分叉。

因此本次设计必须优先保证：

- 列表读模型单一来源
- 购买状态统一计算
- 购买配置有真实写路径
- 代理修改后的重登流程复用已有编排

## 12. 实施结论

本次采用“交互式主列表 + 轻量弹框 + 行右键菜单”的账号中心重构方案。

落地原则如下：

- 列表成为主操作区，不再依赖右侧常驻详情栏
- 四列固定为 `C5昵称 / API Key / 购买状态 / 代理`
- `购买状态` 按 `未登录 > 禁用 > 库存已满 > 当前仓库内容` 统一显示
- `添加账号` 位于列表表头右上角
- 删除账号走行右键菜单并带确认
- 备注、API Key、代理复用现有账号更新接口
- 购买配置新增真实后端命令以支持禁用购买和切换当前仓库

这套方案既能解决当前桌面端账号中心难用的问题，也为将来需要切到浏览器式本地前端时预留了清晰的数据和交互边界。
