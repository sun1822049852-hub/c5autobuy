# 查询启动前商品详情刷新设计

## 1. 目标

把旧版 `autobuy.py` 里“扫货准备页面”的商品价格/磨损刷新逻辑迁入新架构，并保持核心语义一致：

- 进入查询启动前准备步骤时，自动检查商品详情是否过期
- 只刷新超过 12 小时未同步，或从未同步过的商品
- 刷新内容只包含商品详情字段，不改用户设置的查询阈值
- 刷新时使用的账号选择，与“新增商品时补全商品信息”共享同一个全局轮询进度
- 准备步骤中允许用户手动重新刷新，再决定是否真正启动查询

本次设计不改 C5 接口语义，不改查询运行时三模式调度逻辑，不把商品详情刷新塞进查询 worker。

## 2. 源代码逻辑锁定

旧版逻辑不是“点开始就直接扫”，而是：

1. 用户执行扫货
2. 先进入 `show_scan_preparation_page`
3. 进入页面时自动检查每个商品的 `lastModified`
4. 仅刷新：
   - 没有更新时间的商品
   - 距今超过 12 小时的商品
5. 刷新字段：
   - `minwear`
   - `maxwear`
   - `minPrice`
   - `lastModified`
6. 刷新结束后，页面展示结果，并允许：
   - 编辑商品
   - 重新更新所有价格
   - 确认开始扫货

也就是说，旧版的“动态更新”本质上是：

- 启动前自动预刷新
- 准备页里支持手动重刷

不是后台定时任务，也不是查询运行中周期刷新。

## 3. 当前新架构的缺口

当前新架构只实现了“新增商品时抓一次详情并落库”：

- `last_market_price`
- `min_wear`
- `max_wear`
- `last_detail_sync_at`

但缺少以下能力：

1. 没有“按 12 小时阈值筛选待刷新商品”的逻辑
2. 没有批量刷新商品详情的服务
3. 没有查询启动前准备步骤
4. 没有“重新刷新价格”的手动入口
5. `last_detail_sync_at` 目前只在新增商品时写入，后续没有消费

## 4. 方案比较

### 方案 A：启动查询前增加“准备步骤”，自动刷新一次，并支持手动重刷

- 点击“启动查询”后，不直接调用 `/query-runtime/start`
- 先进入准备流程
- 后端返回刷新摘要
- 用户确认后再真正启动查询

优点：

- 最贴近旧版 CLI 逻辑
- 自动刷新和手动重刷都能保留
- 用户能看到哪些商品更新了、哪些失败了
- 不把“带副作用的刷新”偷偷塞进纯启动接口

缺点：

- 需要补一个准备对话框和一个新后端接口

### 方案 B：`/query-runtime/start` 内部自动刷新，然后直接启动

- 前端无变化
- 后端启动前先刷新，随后直接启动

优点：

- 改动最少

缺点：

- 不符合旧版“准备页面 + 用户确认”的流程
- 用户看不到刷新结果
- 无法自然提供“重新刷新价格”
- `start` 变成带较重副作用的接口

### 方案 C：后台周期任务自动刷新商品详情

- 独立定时刷新，不依赖启动查询

优点：

- 商品详情可能更实时

缺点：

- 明显偏离旧版逻辑
- 会引入额外状态和后台任务
- 用户并没有要求运行中自动刷新

## 5. 结论

采用方案 A。

原因很直接：

- 最贴近你源码的真实流程
- 同时保留“自动刷新一次”和“手动重新刷新”
- 边界更清楚：准备是准备，启动是启动

## 6. 新设计

### 6.1 后端新增“查询启动准备”能力

新增一个专门的准备 use case / service，例如：

- `PrepareQueryRuntimeUseCase`
- `QueryItemDetailRefreshService`

职责：

- 读取指定 `QueryConfig`
- 扫描其中全部商品
- 判断哪些商品需要刷新
- 逐个刷新商品详情
- 汇总返回刷新结果

它不负责：

- 真正启动查询 runtime
- 修改查询阈值
- 修改模式配置

### 6.2 刷新资格规则

每个商品按以下规则判断：

需要刷新：

- `last_detail_sync_at` 为空
- `last_detail_sync_at` 无法解析
- 距当前时间 `>= 12 小时`

跳过刷新：

- `last_detail_sync_at` 可解析，且距当前时间 `< 12 小时`

手动重刷时：

- 允许前端传 `force_refresh=true`
- 强制刷新当前配置内全部商品

### 6.3 刷新字段范围

本次只刷新旧版 `ProductDetailCollector` 覆盖的字段：

- `min_wear`
- `max_wear`
- `last_market_price`
- `last_detail_sync_at`

明确不刷新：

- `max_price`
- 用户手工设置的 `max_wear`
- `itemSetName`
- `rarityName`
- `grade`

这里保留当前新架构的字段模型，不在本次扩表。

### 6.4 账号轮询共享规则

这是本次的关键约束：

- 新增商品详情补全
- 查询启动前商品详情刷新

两者必须共用同一个 `DetailAccountSelector` 实例和同一份轮询状态。

实现方式：

- 在 `create_app()` 中只创建一个全局 `DetailAccountSelector`
- `ProductDetailFetcher` 和新的刷新服务都依赖这个同一个 selector
- 不把轮询状态落库
- 只在进程内存中保存当前轮询位置

这样才能满足：

- 这次新增商品用到账号 A
- 下次刷新商品详情时，应从账号 B 开始

而不是每条链各自从头轮询。

### 6.5 刷新执行流程

单次准备刷新流程：

1. 加载指定配置
2. 筛出需要刷新的商品列表
3. 按配置里的商品顺序逐个刷新
4. 每刷新一个商品：
   - 调用已有 `ProductDetailCollector`
   - collector 内部通过共享 selector 获取尝试顺序
   - 某账号失败则切下一个账号
5. 成功则更新该商品详情字段
6. 失败则记录失败信息，但不终止整个批次
7. 返回汇总摘要

这与旧版一致：

- 单个商品失败，不拖垮全部准备流程
- 用户仍可看到失败项，再决定是否继续启动

### 6.6 Repository 变更

当前 `QueryConfigRepository.update_item()` 只改用户阈值字段，不适合承担详情刷新。

因此新增一个更明确的方法，例如：

- `update_item_detail(...)`

职责：

- 只更新商品详情同步字段
- 不覆盖用户设置的 `max_price`
- 不覆盖用户手工修改的 `max_wear`

更新内容：

- `item_name`：保留现有值或按 collector 新值更新
- `market_hash_name`：保留现有值或按 collector 新值更新
- `min_wear`
- `last_market_price`
- `last_detail_sync_at`

对 `max_wear` 的处理要特别注意：

- 配置中的 `max_wear` 现在表示“用户接受的最高磨损”
- 旧版动态刷新只更新商品的完整磨损范围，不会覆盖用户设定阈值

因此本次刷新：

- 不更新 `QueryItem.max_wear`

如果以后要保存“商品天然最大磨损”，需要新增字段，本次不做。

### 6.7 前端准备对话框

当前前端点击“启动查询”会直接调用 `/query-runtime/start`。

本次改成：

1. 点击“启动查询”
2. 打开“启动前准备”对话框
3. 自动调用准备接口
4. 展示结果：
   - 已更新商品数
   - 已跳过商品数
   - 失败商品数
   - 每个商品的状态行
5. 用户可选：
   - `重新刷新价格`
   - `确认启动查询`
   - `取消`

这样与旧版 CLI 准备页语义一致，只是换成 GUI 对话框。

### 6.8 API 设计

新增一个准备接口，例如：

- `POST /query-runtime/prepare`

请求：

- `config_id`
- `force_refresh`，默认 `false`

响应：

- `config_id`
- `config_name`
- `threshold_hours`
- `updated_count`
- `skipped_count`
- `failed_count`
- `items`

每个 item 行至少包含：

- `query_item_id`
- `external_item_id`
- `item_name`
- `status`：`updated` / `skipped` / `failed`
- `message`
- `last_market_price`
- `min_wear`
- `last_detail_sync_at`

现有 `POST /query-runtime/start` 保持职责纯粹：

- 只负责启动
- 不再内部做详情刷新

### 6.9 启动语义

用户在准备对话框中点击“确认启动查询”后：

- 前端再调用现有 `POST /query-runtime/start`
- 后端按当前配置和账号状态启动 runtime

也就是说：

- “准备” 与 “启动” 分成两个显式步骤
- 更符合旧版源码流程

## 7. 错误处理

### 7.1 配置不存在

- 准备接口返回 404

### 7.2 没有可用于详情刷新的账号

- 准备接口返回冲突错误
- 提示没有可用的已登录 token 账号

### 7.3 单商品刷新失败

- 不中断整个准备批次
- 记录失败项
- 允许用户看到失败摘要后决定是否继续启动

### 7.4 全部商品都失败

- 准备接口仍返回摘要
- `failed_count == 商品总数`
- 前端默认不自动禁止“确认启动”，但要明确提示风险

原因：

- 这更贴近旧版 CLI：刷新失败并不强行终止后续流程

## 8. 测试重点

本次必须锁定这些行为：

1. 超过 12 小时的商品会被筛入刷新列表
2. 12 小时内的商品会被跳过
3. `force_refresh=true` 会强制刷新全部商品
4. 刷新时和新增商品共享同一个 selector 轮询进度
5. 单商品失败不会中断整批准备
6. 刷新成功只更新详情字段，不覆盖用户阈值字段
7. 准备接口返回的摘要结构稳定
8. 前端点击“启动查询”时先进入准备步骤，而不是直接启动

## 9. 非目标

本次不做：

- 查询运行中周期刷新商品详情
- 后台定时任务自动刷新
- 恢复 `itemSetName / rarityName / grade`
- 扩表保存“商品天然最大磨损”
- 多配置并发预刷新

## 10. 结论

这次按源码真实流程落地：

- 启动查询前先进入准备步骤
- 自动按 12 小时阈值刷新商品详情
- 允许手动重新刷新
- 新增商品与准备刷新共享同一个全局轮询账号选择器

这样既补回旧版关键行为，又不把详情刷新掺进查询 runtime 内部，让新架构边界继续保持清楚。
