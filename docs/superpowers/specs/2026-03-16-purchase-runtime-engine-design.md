# 购买运行时设计

## 1. 目标

在现有前后端分离重构架构上，补齐购买链路，使“查询命中 -> 购买调度 -> 账户执行 -> 库存回写 -> 运行态展示”形成独立闭环，同时保持以下原则不变：

- UI 只负责输入、展示、控制，不直接操作购买逻辑
- 查询和购买通过命中事件解耦，不把购买逻辑塞回查询模块
- 保留 legacy 的核心行为，不为了结构好看改掉实际运行逻辑
- 购买能力与购买池状态继续分离

本阶段的目标不是一次性重写全部下单细节，而是先建立稳定边界和可验证的购买运行时。

## 2. 已确认的 legacy 行为

### 2.1 购买任务来源

购买任务只来自查询命中结果，不引入新的手工购买任务入口。

legacy 入口链路是：

- 查询命中回调 `_on_query_result`
- 提交到 `scheduler.submit_batch`
- 调度器把批次分发到账户购买工作池

因此新架构里也必须保留“查询命中驱动购买”的单一入口。

### 2.2 购买能力与购买池状态

购买能力与购买池状态必须严格区分：

- 有效登录态 / token，表示账户具备购买能力
- 库存不足，只表示账户暂时不能继续参与购买池轮询
- 库存不足时，账户会被移出购买池，但不会失去购买能力

### 2.3 仓库库存运行逻辑

仓库逻辑按 legacy 实际行为保留：

1. 已登录账户进入购买链路初始化时，先获取一次仓库快照
2. 根据仓库快照计算每个小仓库的剩余容量
3. 只保留满足阈值的可用仓库
4. 在可用仓库中，优先选择“剩余容量更小但仍满足阈值”的目标仓库
5. 每次购买成功后，优先根据购买成功数量在本地递推仓库占用
6. 如果当前目标仓库不足，则尝试切换到下一个本地可用仓库
7. 只有本地可用仓库耗尽时，才重新拉取最新仓库快照确认
8. 确认无可用仓库后，将账户移出购买池，并启动恢复检查
9. 恢复检查确认仓库恢复后，再把账户放回购买池

这里保留 legacy 的一个重要现实：

- 仓库快照会落盘保存
- 但它不是启动时唯一真相源
- 购买运行时启动后仍会主动刷新一次最新仓库状态

### 2.4 全局购买过滤规则

以下规则是全局生效，不挂到账户独立配置里：

- `仅查询模式`
- `购买账号白名单`

购买资格最终由以下条件共同决定：

- 当前不是 `仅查询模式`
- 账户 `purchase_capability_state == bound`
- 账户未被禁用
- 白名单为空，或账户命中白名单
- 账户当前存在可用仓库

## 3. 范围与非目标

### 3.1 本阶段范围

- 建立新购买运行时
- 建立命中事件到购买链路的桥接
- 建立账户购买池调度
- 建立库存快照、目标仓库、本地容量递推和恢复检查
- 建立运行态 API 与基础 GUI
- 建立最近购买事件和运行摘要

### 3.2 本阶段非目标

- 不新增手工购买任务入口
- 不重做登录流程
- 不改变用户手动扫码 + 浏览器抓取信息的原登录逻辑
- 不改变查询模块的三模式时间窗设计
- 不在 UI 层实现购买判断

## 4. 分层方案

### 4.1 PurchaseRuntimeService

职责：

- 启动购买运行时
- 停止购买运行时
- 返回运行摘要和最近事件
- 接收查询模块推送的命中批次

它是后端对外总入口，但不直接负责下单细节。

### 4.2 PurchaseHitInbox

职责：

- 接收查询命中批次
- 做短时间去重
- 把通过的批次放入购买调度队列

输入结构沿用查询命中现有字段，例如：

- `query_item_name`
- `product_list`
- `total_price`
- `total_wear_sum`
- `account_display_name`
- `mode_type`

去重策略保持 legacy 思路，以 `total_wear_sum` 为核心键，加短时间窗口。

### 4.3 PurchaseScheduler

职责：

- 维护全局待购买队列
- 维护当前可参与购买的账户池
- 轮询选择账户
- 把批次分发给账户工作者

它只关心“谁现在可参与轮询”，不关心具体怎么下单。

### 4.4 AccountPurchaseWorker

职责：

- 一账户一工作池
- 接收调度器分发的批次
- 调用执行网关完成下单
- 回写执行结果
- 触发库存递推和状态同步

### 4.5 PurchaseExecutionGateway

职责：

- 封装订单创建、支付、重试、防重复等执行细节
- 初期允许以 legacy-compatible 适配方式接入
- 把协议级细节隔离在边界内

这样后续替换下单实现时，不会污染调度器和运行时。

### 4.6 PurchasePoolStateSync

职责：

- 维护账户购买池状态
- 维护库存不足、鉴权失效、恢复入池
- 维护最近错误与最近购买事件
- 同步运行态摘要

## 5. 数据建模

### 5.1 账号主模型保持精简

`accounts` 主表继续只承载稳定账号信息：

- 账号名称、备注、代理、API key
- 登录态相关信息
- 查询模式开关
- `purchase_capability_state`
- `purchase_pool_state`

不把高频波动的仓库快照直接塞进账号主表。

### 5.2 新增账户库存快照附属存储

新增与账号强绑定的库存快照存储，例如：

- `account_inventory_snapshots`

建议字段：

- `account_id`
- `selected_steam_id`
- `inventories_json`
- `refreshed_at`
- `last_error`

这份数据在业务上属于账号，但在结构上独立于主账号表。

这里的 `selected_steam_id` 不是展示字段，而是当前目标仓库标识：

- 购买执行时会把它对应的仓库作为下单目标
- 对应 legacy 下单请求体中的 `receiveSteamId`
- 账户工作者执行购买时必须能拿到当前选中的目标仓库

也就是说，库存快照里至少要同时保存两类信息：

- 当前可用仓库集合
- 当前选中的目标仓库

### 5.3 新增购买运行设置存储

新增购买运行设置存储，例如：

- `purchase_runtime_settings`

建议字段：

- `query_only`
- `whitelist_account_ids_json`
- `updated_at`

这是全局设置，不挂到账户上。

### 5.4 最近购买事件

最近购买事件可以先以内存环形缓冲为主，必要时再补持久化。

事件字段建议包括：

- `event_id`
- `occurred_at`
- `event_type`
- `account_id`
- `account_display_name`
- `query_item_name`
- `product_list`
- `total_price`
- `total_wear_sum`
- `message`
- `status`

## 6. 库存子系统设计

### 6.1 初始化

购买运行时启动时：

1. 读取账户库存快照
2. 为具备购买能力的账户创建库存选择器
3. 主动刷新一次最新仓库快照
4. 计算可用仓库列表
5. 选择目标仓库
6. 根据结果决定是否进入购买池

快照只用于辅助展示和兜底，不替代初始化刷新。

初始化完成后，运行时必须把当前目标仓库写回 `selected_steam_id`，以便：

- 运行态展示当前目标仓库
- 账户工作者执行购买时使用正确的目标仓库
- 后续本地容量递推和切仓有稳定锚点

### 6.2 可用仓库筛选

沿用 legacy 行为：

- 计算 `remaining = inventory_max - inventory_num`
- 只保留满足容量阈值的仓库

目标仓库选择保持 legacy 实际行为：

- 优先选择当前占用更高、也就是剩余容量更小但仍满足阈值的仓库

### 6.3 购买后的本地递推

每次购买成功后：

1. 根据返回的购买成功数量更新当前目标仓库本地占用
2. 重新计算剩余容量
3. 若当前仓库仍可用，继续使用
4. 若当前仓库不可用，则尝试切换到下一本地可用仓库
5. 若本地可用仓库已空，再拉一次最新仓库快照确认

每一次“继续沿用当前仓库”或“切换到新仓库”，都必须同步更新 `selected_steam_id`。

### 6.4 无库存与恢复

确认无库存后：

- 账户从购买池移出
- `purchase_pool_state` 更新为 `paused_no_inventory`
- 启动恢复定时器

恢复检查成功后：

- 刷新快照
- 重新选定目标仓库
- 账户回到购买池
- `purchase_pool_state` 更新为 `active`

## 7. 运行时数据流

1. 查询模块产生命中事件
2. 命中事件进入 `PurchaseHitInbox`
3. `PurchaseHitInbox` 去重后写入 `PurchaseScheduler`
4. `PurchaseScheduler` 从可用购买池轮询选账户
5. 任务分发给对应 `AccountPurchaseWorker`
6. `AccountPurchaseWorker` 调用 `PurchaseExecutionGateway`
7. 执行结果回写 `PurchasePoolStateSync`
8. 库存递推、切仓、无库存确认、恢复检查在库存子系统中完成
9. `PurchaseRuntimeService` 汇总运行态，供 API 和 GUI 展示

## 8. 状态机

### 8.1 购买能力状态

- `unbound`：无有效登录态，不能购买
- `bound`：具备购买能力
- `expired`：运行时确认鉴权失效，需要重新登录

### 8.2 购买池状态

- `not_connected`：未接入购买运行时
- `active`：在购买池中，可参与轮询
- `paused_no_inventory`：库存不足，已移出购买池
- `paused_auth_invalid`：鉴权失效，已移出购买池

### 8.3 关键迁移

- 登录成功：`unbound -> bound`
- 运行时接入成功：`not_connected -> active`
- 库存不足：`active -> paused_no_inventory`
- 库存恢复：`paused_no_inventory -> active`
- 鉴权失效：能力 `bound -> expired`，池状态转 `paused_auth_invalid`
- 停止运行时：池状态统一回 `not_connected`

## 9. API 与 GUI 最小闭环

### 9.1 后端 API

建议最小接口：

- `GET /purchase-runtime`
- `POST /purchase-runtime/start`
- `POST /purchase-runtime/stop`
- `GET /purchase-runtime/settings`
- `PUT /purchase-runtime/settings`
- `GET /purchase-runtime/events`

其中：

- `settings` 负责 `仅查询模式` 与 `购买账号白名单`
- 运行状态由 `purchase-runtime` snapshot 返回

### 9.2 GUI 布局

新增独立“购买运行页”，不把运行控制塞回账号中心。

页面包含：

- `开始购买`
- `停止购买`
- `仅查询模式`
- `购买账号白名单`
- 运行摘要
- 账户池表格
- 最近购买事件

账号中心继续只展示：

- 购买能力
- 购买池状态
- 最近错误

不在账号中心里做购买运行控制。

### 9.3 查询页联动提示

查询运行页可以补一个轻量联动提示：

- `已转入购买`
- `仅查询模式已拦截`

这样用户能直接看到命中事件是否进入购买链路。

## 10. 异常处理

### 10.1 命中重复

- `PurchaseHitInbox` 丢弃短时间重复批次

### 10.2 当前无可用购买账户

- 批次保留在待处理队列
- 运行态显示“等待可用购买账户”

### 10.3 单账户执行失败

- 只记录该账户事件和错误
- 不拖垮整体购买运行时

### 10.3 当前目标仓库丢失

- 如果账户存在库存快照但没有当前 `selected_steam_id`
- 或 `selected_steam_id` 对应仓库已不在当前可用集合中
- 不允许继续盲目下单
- 必须先重新选择目标仓库，必要时刷新一次远端仓库快照

### 10.4 库存确认失败

- 本地耗尽后先尝试远端确认
- 如果确认失败，按 legacy 行为把账户视为暂不可用

### 10.5 鉴权失效

- 从购买池移出
- 更新 `purchase_capability_state = expired`
- 运行态与账号详情均提示需要重新登录

## 11. 测试策略

### 11.1 后端单测

- `PurchaseHitInbox` 去重逻辑
- `PurchaseScheduler` 轮询分发
- 白名单过滤
- `仅查询模式` 拦截
- 购买后本地容量递推
- 本地切仓
- 本地耗尽后远端刷新
- 无库存移池但不掉购买能力
- 鉴权失效转 `expired`

### 11.2 后端集成

- 查询命中进入购买链路
- 运行时启动后账户初始化入池
- 购买成功后库存快照回写
- 恢复检查后重新入池
- 运行摘要与最近事件接口正确返回

### 11.3 前端测试

- 购买运行页启停
- 全局设置提交
- 账户池状态展示
- 最近购买事件展示
- 账号中心状态联动刷新

## 12. 实施顺序建议

1. 先建立购买运行时骨架与 API
2. 再接命中事件入口与调度器
3. 然后实现库存快照、目标仓库、本地递推和恢复检查
4. 再接执行网关
5. 最后补 GUI 和测试

这样能先把结构立住，再逐步把 legacy 行为迁进来，而不是继续让购买逻辑散落在一个大文件里。
