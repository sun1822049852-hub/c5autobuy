# 查询运行时全局商品调度设计

## 1. 目标

在当前前后端分离架构下，把查询运行时改成更贴近 legacy 的执行模型：

- 一个查询配置包含多个商品
- 三种查询模式共用这一组商品
- 商品分配由配置级全局调度器完成
- 每个“账号 + 模式”独立进入冷却并独立申请下一个商品

本次改动保持以下边界不变：

- 仍只允许一个查询任务同时运行
- UI 只负责配置输入、状态展示、启动停止
- scanner 协议细节继续复用 legacy
- 命中结果仍可转发给购买运行时

## 2. legacy 真实分配逻辑

legacy 不是“每个账号每轮把所有商品查一遍”，也不是“每个模式自己维护商品轮转”。

真实逻辑是：

1. 启动时把当前配置下所有商品组成 `product_pool`
2. 每个可用的查询组在冷却结束后通知全局调度器“我已就绪”
3. 全局调度器从统一商品池里按指针分配一个商品给该查询组
4. 查询组只执行这一个商品
5. 查询完成后该查询组重新进入自己的冷却

这意味着：

- 商品分配是配置级全局行为
- 冷却与时间窗口是账号模式级行为
- 查询粒度是“一次就绪 -> 一个商品”

## 3. 当前重构版的问题

当前新运行时虽然已经能调用 legacy scanner，但执行粒度仍偏离 legacy：

- `ModeRunner` 以“模式”为中心统一跑一轮
- 一轮中会给多个 worker 各分一个商品
- 商品轮转在 `ModeRunner` 内部完成

这样的问题是：

- 商品分配不再是配置级全局共享
- 同一模式下多个账号被人为绑成同一节奏
- 账号冷却和商品分配的职责边界混在一起

## 4. 新设计

### 4.1 QueryTaskRuntime

职责：

- 持有当前 `QueryConfig`
- 创建配置级 `QueryItemScheduler`
- 为每个模式创建 `ModeRunner`
- 聚合模式统计和最近事件

它不负责：

- 单账号查询执行
- scanner 细节
- 商品查询协议

### 4.2 QueryItemScheduler

新增配置级商品调度器，职责：

- 持有配置下全部 `QueryItem`
- 维护全局商品指针
- 维护商品最小间隔
- 当某个账号模式 worker 就绪时，为其分配一个商品和计划执行时间

行为约束：

- 一次只分配一个商品
- 同一商品的连续调度要满足最小冷却
- 所有模式共用同一个商品池

### 4.3 ModeRunner

保留“模式级容器”角色，但不再做商品轮转。

职责变为：

- 按模式筛选符合条件的账号
- 为每个符合条件账号创建一个 worker task
- 共享本模式的时间窗口与冷却配置
- 聚合本模式统计和事件

它不再负责：

- 决定下一个商品是谁
- 让所有账号共用同一轮节奏

### 4.4 AccountQueryWorker

继续负责：

- 单账号单模式执行一次查询
- 调用 legacy scanner
- 处理 403 / 429 / Not login / 普通错误
- 输出统一事件

新增配合方式：

- worker loop 在“准备就绪”时向 `QueryItemScheduler` 申请一个商品
- 完成一次查询后，按当前模式配置重新进入独立冷却

## 5. 数据流

### 5.1 启动

1. `QueryRuntimeService.start(config_id)`
2. 创建 `QueryTaskRuntime`
3. `QueryTaskRuntime` 创建一个 `QueryItemScheduler`
4. 为 `new_api / fast_api / token` 分别创建 `ModeRunner`
5. 每个 `ModeRunner` 为自己模式下的符合条件账号启动独立 worker loop

### 5.2 单次执行

1. worker 检查当前模式是否启用
2. worker 检查当前是否在该模式的时间窗口内
3. worker 冷却结束后向全局 `QueryItemScheduler` 申请商品
4. 调度器返回一个商品和计划执行时间
5. worker 在计划时间执行该商品查询
6. 返回事件并更新统计
7. 若命中则转发给购买运行时
8. worker 重新进入自己的下一轮冷却

## 6. 状态模型

现有前端面板可继续消费任务级和模式级状态，字段保持兼容，重点是语义修正：

- `eligible_account_count`: 当前模式下符合资格的账号数
- `active_account_count`: 当前模式下仍有运行能力的账号数
- `query_count`: 实际执行的查询次数
- `found_count`: 实际命中总数
- `recent_events`: 所有模式聚合后的最近事件

不新增前端强耦合字段，先保证后端行为正确。

## 7. 错误处理

- 403 / `Not login`: 仅禁用对应账号在对应模式下的查询
- 429: 对应账号进入退避冷却，不影响其他账号
- 时间窗口关闭：worker 等待到下一次窗口开始
- 商品池为空：所有模式空转但不崩溃
- 单 worker 异常：记录事件，不拖垮整个任务

## 8. 测试重点

本次必须新增或修正的测试：

1. 配置级商品调度器按全局顺序分配商品
2. 同一商品满足最小冷却后才再次分配
3. 同一模式下多个账号独立申请商品，而不是统一一轮执行
4. 查询命中仍能转发购买运行时
5. 原有时间窗口、资格筛选、403/429/未登录处理不回归

## 9. 非目标

本次不做：

- WebSocket / SSE 实时日志
- 查询结果持久化入库
- 多查询配置并发运行
- 移除 legacy scanner
